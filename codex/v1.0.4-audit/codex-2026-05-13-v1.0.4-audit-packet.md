## What I am uncertain about

1. **Cross-task interactions.** Task 5's MCS resolver against Task 4's plate guard against Task 3's nozzle guard, on combinatorial state (multi-color + non-stocked nozzle + non-stocked plate + cold environment). Walkthrough covers each independently; do they interact correctly in combination?
2. **Asymmetric helper API surface.** Task 3 shipped `getCompatibleNozzlesForPrinter`; Task 4 deferred the parallel `getCompatiblePlatesForPrinter`. Reviewer-flagged in S3, controller-deferred per no-relitigation. Is this acceptable for v1.0.4 or worth resolving here?
3. **`_mcsTier` engine-surface exposure.** Internal-only after Task 5 (reviewer noted in S4); future UI consumption (Phase 3 / v1.0.5) may want it on the public API. Should v1.0.4 expose it preemptively or wait for a consumer?
4. **Warning-ID taxonomy coherence.** Phase 1 introduced 8 new warning IDs (Task 1 emits no new warning). Are naming conventions consistent across `*_not_on_printer`, `mcs_tier_*_guidance`, `nozzle_below_min_diameter`, `chamber_above_material_safe`, `env_compensation_capped_by_material`, `ams_lite_material_incompat`, `mcs_empty_no_multicolor`? Are fix-text quality and severity calibrated correctly across the new set?
5. **Test-contract coverage.** Walkthrough adds 7 positive blocks (9 sub-assertions in MCS); matrix-audit's 47196 broad-config sweep covers the negative space. Are there orthogonal-state combos missing? Specifically: does the broad sweep exercise multi-color × env × printer × material in combinatorial space, or is each axis swept independently?
6. **Bambu export-path deferral risk.** Task 2 surfaced that `exportBambuStudioJSON` reads raw `material.base_settings.cooling_fan_min` (not the env-multiplied value) and lacks a `BAMBU_PROCESS_MAP` entry for `draft_shield`. Filed under ROADMAP "IR-deferred export-path findings" with a v1.0.4-specific note. Is shipping v1.0.4 with this known gap acceptable while export UI is disabled?
7. **Data file consistency.** Did all printers / materials get the new fields uniformly (`available_nozzle_sizes`, `available_plates`, `ams_lite_compatible`, `enclosure_behavior.safe_chamber_temp_max`, etc.), or only the ones positive cases touched? Check for partial coverage that could cause silent regressions in untested combos. Specifically: only 4 materials (`petg_basic`, `petg_hf`, `petg_cf`, `pet_cf`) carry `safe_chamber_temp_max` — is the absence-implies-no-cap interpretation safe in the chamber-guard code path?
8. **Retired warning-ID matrix-audit sweep.** Tasks 5 and 7 retired or renamed warning IDs (`k2_plus_cfs` → `mcs_tier_cfs_guidance`; `cf_small_nozzle` retired; `nozzle_too_small` → `nozzle_below_min_diameter`). Did the rename/retire sweep cover ALL of `scripts/*` — walkthrough, matrix-audit, validate-data, any other test/audit files? Are stale references to retired IDs still present anywhere? (S5 caught one drift in `profile-matrix-audit.js:280` pre-commit; was anything missed?)
9. **Walkthrough harness negative assertions.** Per S2-learned discipline, the harness uses exact-ID `.includes('warning_id_exact')` for positive assertions. Does it include negative assertions (`!warnings.some(w => w.id === '<retired_id>')`) for retired IDs, ensuring they don't reappear? Positive-only coverage misses test-contract drift.
10. **Subsume-vs-coexist decision on Task 7.** S5 retired `cf_small_nozzle` entirely and renamed `nozzle_too_small` → `nozzle_below_min_diameter` with a generic parameterized body. Both originally fired on the same condition (`nozzle.size < material.nozzle_requirements.min_diameter`). The subsume decision was correct, but does the parameterized body lose any information that the dual-warning approach carried (e.g., CF-specific advice that PLA users don't need)?
