# v1.0.4 config-impact audit response

## Findings

### HIGH-01

Live web UI/export surfaces still bypass the env-aware fan/draft-shield outputs.

Evidence:

- `engine.js:1243-1317` adds env-scaled `fan_max_speed` / `fan_min_speed` S-wrapped fields, but the same block keeps legacy `cooling_fan_min` unscaled "by design" at `engine.js:1268-1274`.
- `app.js:1244-1248` renders `adv.cooling_fan_min`, `adv.cooling_fan_overhang`, and `adv.slow_layer_time`; it does not render `adv.fan_min_speed` or `adv.fan_max_speed`. `engine.js:3131-3139` has the same issue in text export.
- `engine.js:2674-2678` has an explicit TODO that `draft_shield -> enable_draft_shield` is missing from Bambu export mapping. `engine.js:3046-3050` has an explicit TODO that `env.fan_multiplier` is not applied to Bambu filament export.
- The packet treats export deferral as low-risk because export UI is disabled, but `app.js:814-846` binds live process, filament, and copy export buttons; `app.js:989-1006` shows the Bambu export group for Bambu slicer printers and the copy export for non-Bambu printers; `index.html:127-132` contains those controls.

Recommendation:

Before iOS sync / v1.0.4 ship, either wire the live web surfaces to the env-aware fields and export mappings, or explicitly disable affected export paths. Minimum concrete fix: app UI and text export should read the env-scaled fan values; Bambu process export should map `draft_shield`; Bambu filament export should use the same env-scaled fan min/max values the engine emits. Add one audit assertion against the live export payload, not only against the new S-wrapped engine fields.

### HIGH-02

Cold-environment and active-chamber guidance can tell PLA users to do the unsafe thing.

Evidence:

- PLA data says the material needs an open-door heat-creep mitigation: `data/materials.json:40-44` has `open_door_threshold_bed_temp: 45` and the reason "PLA softens at 57degC"; `data/materials.json:88-89` records heat resistance as 57degC.
- X1E is active-heated to 60degC: `data/printers.json:163-165`; its notes still say to open the door/top for PLA at `data/printers.json:177-180`.
- `getChecklist` adds "Preheat enclosure" for any enclosed printer in cold/vcold environments at `engine.js:1366-1378`, independent of PLA heat-creep/open-door behavior.
- The cold environment's second warning says "Keep door closed throughout print" at `data/rules/environment.json:31-34`. `getWarnings` rewrites only the first env warning and preserves subsequent JSON warnings verbatim at `engine.js:1607-1612` and `engine.js:1641-1643`.
- The PLA heat-creep warning only fires for `printer.enclosure === 'passive'` at `engine.js:1502-1507`, so active-heated PLA does not get it. The chamber safe-cap guard only fires when `safe_chamber_temp_max` exists at `engine.js:1746-1749`, and the walkthrough harness codifies X1E + PLA silence at `scripts/walkthrough-harness.js:849-855`.

Recommendation:

Make cold/chamber copy material-aware before shipping. Either add `safe_chamber_temp_max` or an equivalent active-chamber cap for PLA-family materials, or derive it from `open_door_threshold_bed_temp` / heat-resistance data. Do not tell PLA users to keep an enclosure closed; for PLA on enclosed/active-heated printers, prefer explicit "door/top open" guidance or suppress the generic cold-enclosure preheat item. Remove the harness assertion that X1E + PLA must stay silent.

### MEDIUM-01

First-layer bed compensation can claim a delta that was fully clipped by clamps.

Evidence:

- `getAdvancedFilamentSettings` adds `env.bed_first_layer_adj` into `initBed` at `engine.js:1227`, then clamps `initBed` to `bedCap` at `engine.js:1234-1238`.
- `getWarnings` emits "`+NdegC applied to first-layer bed temperature`" whenever `env.bed_first_layer_adj > 0` at `engine.js:1624-1631`; it does not calculate the effective post-clamp delta.
- PETG Basic is a concrete static example: `data/materials.json:520-526` has `bed_temp_base: 75`, `bed_temp_max: 85`, and `initial_layer_bed_offset: 5`; because PETG gets the engine's additional +5 at `engine.js:1227`, the normal initial bed already reaches the 85degC material cap. In cold env, `data/rules/environment.json:25` asks for another +7degC, but the emitted value remains clamped at 85degC while the warning still says +7degC was applied.

Recommendation:

Compute and report the effective post-clamp first-layer bed delta. If the requested delta is partially or fully clipped, either emit a cap/clip warning or change the copy to "requested but clipped." Also include `bed_first_layer_adj` in the bed-clamp warning calculation around `engine.js:1706-1727`, so bed compensation has the same attribution discipline as nozzle compensation.

### MEDIUM-02

The broad matrix audit does not sweep the axes most of v1.0.4 changed.

Evidence:

- `scripts/profile-matrix-audit.js:51-61` fixes `colors: 'single'`, `environment: 'normal'`, and has no `build_plate` in `DEFAULT_STATE`.
- `scripts/profile-matrix-audit.js:482-499` builds the broad sweep across only printer, material, nozzle, speed, and profileMode.
- Therefore the claimed 47196 clean configs do not combinatorially exercise multicolor/MCS warnings, cold/vcold env compensation, build-plate inventory/range guards, or support/build-plate cross-effects. The report text at `scripts/profile-matrix-audit.js:708-709` accurately describes the narrower axes; the review packet over-reads it as broader negative-space coverage.

Recommendation:

Add targeted pairwise or small-axis sweeps for the new release axes: `colors` (`single`, `multi_2_4`, `multi_5`), `environment` (`normal`, `cold`, `vcold`), and representative `build_plate` values per printer/material. If the full product is too large, keep the current broad sweep but rename its release claim so it does not imply coverage it does not provide.

### LOW-01

Retired warning IDs are not all guarded against co-firing in successor cases.

Evidence:

- The walkthrough asserts `cf_small_nozzle` is absent at `scripts/walkthrough-harness.js:874-875`, but there is no equivalent negative assertion for retired `nozzle_too_small`.
- CFS successor walkthrough cases assert `mcs_tier_cfs_guidance` at `scripts/walkthrough-harness.js:771-775` and `scripts/walkthrough-harness.js:780-783`, but they do not assert retired `k2_plus_cfs` is absent.
- `scripts/profile-matrix-audit.js:143` checks `noWarning('k2_plus_cfs')` only on the P1S AMS case, not on the K2/K2 Plus successor cases at `scripts/profile-matrix-audit.js:145-146` and `scripts/profile-matrix-audit.js:282-283`.
- `creality_no_multicolor` is guarded in one K2 postfix case at `scripts/profile-matrix-audit.js:282-283`, but not in the generalized empty-MCS walkthrough successor case at `scripts/walkthrough-harness.js:736-745`.

Recommendation:

Add exact negative assertions for every retired ID in the relevant successor positive case: `nozzle_too_small`, `k2_plus_cfs`, and `creality_no_multicolor`, while keeping the existing `cf_small_nozzle` guard. This is test-contract hardening; it is not a release blocker by itself.

### OBSERVATION-01

The helper/API asymmetry is acceptable for v1.0.4 if it is documented as warning-only debt.

Evidence:

- `getCompatibleNozzlesForPrinter` is public at `engine.js:2609-2631` and exported at `engine.js:3401-3403`.
- The parallel plate behavior exists only as a warning-time guard at `engine.js:1835-1848`; there is no public `getCompatiblePlatesForPrinter`.
- `_mcsTier` is internal at `engine.js:962-977`, and the scoped code only uses it from engine-owned warning/checklist/profile paths.

Recommendation:

Do not expose `_mcsTier` preemptively. Add `getCompatiblePlatesForPrinter` only when the web/iOS plate picker is ready to consume it; otherwise explicitly document v1.0.4 plate support as warning-only. This is good enough for v1.0.4 and should not block the release once the higher-severity issues are fixed.

## Summary

- CRITICAL: 0
- HIGH: 2
- MEDIUM: 2
- LOW: 1
- OBSERVATION: 1

Overall recommendation: remediate-then-ship. The core architecture direction is sound, and the MCS/nozzle/plate warning taxonomy is mostly coherent, but v1.0.4 should not proceed to iOS sync while live web export/UI surfaces still bypass env-aware outputs and while PLA cold/chamber guidance can contradict the material safety rules.

Confidence level: medium. I have high confidence in the listed findings because they are static code/data contradictions, but I did not run tests per instruction, so I am not claiming a full clean bill beyond this review scope.
