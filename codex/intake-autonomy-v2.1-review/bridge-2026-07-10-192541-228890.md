### Findings

**[SHOULD-FIX — concrete invariant gap] `taintedAllowed` strict-equality bypass in both the validator and the redirect**

- `classifyParkReason` redirects tainted candidates only when `def.taintedAllowed === false`. A future class entry with `taintedAllowed` omitted (or `null`) and `reviewEntry: true` / timer trigger would **not** be redirected — a tainted candidate would fall through into it.
- `validateTaxonomyGraph` fires violations only when `def.taintedAllowed === true`. The same absent field is invisible to the check.
- Net effect: the "headline regression guard" (spec §2, §7) is bypassable by omitting a single key in a new taxonomy entry. Three spec drafts reintroduced the ratchet; this test exists to catch the fourth. It cannot catch what it doesn't see.
- Fix direction: both functions should treat absent/null `taintedAllowed` as "not explicitly denied" and require default-deny semantics — redirect should use `!def.taintedAllowed` instead of `=== false`; validator violation condition should be `def.taintedAllowed !== false` for any class where that would permit `reviewEntry:true` or a timer trigger.
- The current six-class taxonomy is safe (all entries explicit), so this is not a live defect — but the guard's robustness guarantee fails for any future addition.
