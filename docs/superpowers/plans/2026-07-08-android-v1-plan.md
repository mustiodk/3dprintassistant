# Android v1 — Gated Master Implementation Plan

> **For agentic workers:** execute ONE GATE per session. At gate entry, author that gate's micro-plan at `superpowers:writing-plans` granularity (exact files, failing tests first, code, commands, commits) in `docs/superpowers/plans/android-v1/gate-N-<name>.md`, then implement via `superpowers:executing-plans` / `superpowers:subagent-driven-development`. This master plan deliberately stays at gate granularity (same rationale as the accepted iOS v2 master plan): code-level steps written today for a greenfield repo weeks out would be stale and unreviewable — each micro-plan is written against the repo as it actually exists, then hostile-reviewed before code.

**Goal:** Ship **3D Print Assistant for Android v1.0.0** to Google Play — a native Kotlin/Compose app with the improved-from-day-one bar (full web parity where natively sensible: native Mine learns-loop, troubleshooter, journal, workshop + byte-compat backup, outbound share, estimators, a11y, EN+DA), running `engine.js` + mirrored JS modules byte-identical in an embedded QuickJS runtime.

**Architecture:** Per design spec AD0–AD10 (`docs/superpowers/specs/2026-07-08-android-v1-design.md`). Native restart in the existing `3dprintassistant-android` repo; EngineCore/EngineAPI bridge on a single-thread dispatcher; learns layer byte-identical JS with a Kotlin write-side store; same-shaped Android printer overlay; Play internal→closed→production track ramp.

**Tech stack:** Kotlin, Jetpack Compose (Material 3, brand-themed), QuickJS via vendored `quickjs-wrapper`, Gradle Kotlin DSL (wrapper; no Android Studio dependency), minSdk 29 / targetSdk 36, GitHub Actions (build+instrumented-test per push; manual-dispatch Play upload via `r0adkll/upload-google-play`), Play App Signing.

**Spec:** `docs/superpowers/specs/2026-07-08-android-v1-design.md` · **Audit-delta:** `docs/reviews/2026-07-08-android-v1-audit-delta.md` · **Platform-neutral evidence base:** `docs/reviews/2026-07-08-ios-v2-rebuild-audit.md` (§4 contracts) · **Review dispositions:** `docs/reviews/2026-07-08-android-v1-review.md`

## Standing execution protocol (every gate, every session)

1. **Cold start** per Trigger C (3dpa read order) + read this plan + the gate ledger `docs/planning/ANDROID-V1-GATE-LEDGER.md` (web repo — pushed, cross-machine visible).
2. **Machine prerequisites** (halt and surface if any fail): this is an **execution session → must run on the mac-mini** (spec §4.3; the Air has no toolchain — verified 2026-07-08). Check: Java 17 (`java -version`), Android cmdline tools + build-tools 36 + platform android-36 (`sdkmanager --list_installed` or the prototype's verified path `/usr/local/share/android-commandlinetools`), an API-36 emulator boots (`avdmanager list avd`), **Node.js present (`node --version` — protocol step 4 runs five Node commands; NOT yet verified on the mac-mini, check at AG1 session start)**, the web repo present + current, `3dprintassistant-android` repo present + `git fetch` clean. Planning/doc-only sessions may run anywhere.
3. **Sync sweep (ledger-recorded):** byte-diff mirrored files against web `main` — `engine.js`, `data/materials.json`, `data/nozzles.json`, `data/rules/`, `locales/en.json`, `locales/da.json`, `workshop-tuning.js`, `workshop-tuning-rules.js`, `state-codec.js` (all mirrored at AG1 — the spike loads all four JS files), **and `printers.json` (pre-release only — greenfield bundles the live snapshot; at v1.0.0 the bundled baseline freezes and the overlay model takes over per AD8, at which point printers.json LEAVES this sweep — flip recorded in the ledger).** Any drift → mirror-sync as the gate's FIRST commit (web is master), re-run contract fixtures, record the synced web SHA in the ledger.
4. **Baselines:** full test suite green before touching code — JVM: `./gradlew testDebugUnitTest`; instrumented (emulator): `./gradlew connectedDebugAndroidTest` (pin the exact AVD name in the ledger at AG1). Web-side contract sources (run in the web repo): `node scripts/engine-golden-snapshot.js --check` + `node scripts/state-codec.test.js` + `node scripts/workshop-store.test.js` + `node scripts/workshop-tuning.test.js` + `node scripts/workshop-tuning-rules.test.js` all green (they generate/verify the fixtures Android pins against).
5. **Gate micro-plan** (writing-plans granularity) → hostile fresh-context sub-agent review → patch → then code.
6. **TDD RED-first** per repo convention (incl. TDD-RED breadcrumb rules for degenerate mirror-test REDs — the cross-platform mirror convention applies to Android identically).
7. **String rule (from AG2, greenfield):** every gate adding UI strings adds EN+DA resource entries in the same commit; DA copy flagged in that gate's screenshot packet.
7a. **Fixture-evolution rule:** contract fixtures change ONLY to track web-regenerated fixture sources (after a recorded mirror-sync); any assertion weakening or removal is enumerated in the gate's ledger row (greenfield's analog of the iOS "equivalent-or-stronger" migrated-test rule — the fixture floor gets the same protection the 135-test floor had).
8. **Gate exit:** full suite + contract fixtures green → hostile fresh-context sub-agent review of the gate diff → patch (one finding = one commit) → QA green → tick the gate ledger → commits pushed to `origin/main` (AD1: pre-release main is pushable; it IS the backup).
9. **UI gates additionally:** extend/maintain the screenshot harness (Compose-test/UiAutomator — built at AG2; a broken harness blocks that gate's exit) → produce the packet + a one-page "what changed" note for async owner review; owner feedback lands as patch commits at the next gate boundary (non-blocking).
10. **Cross-model review** (`bridge --health` → `bridge --mode codex-only`, direct `codex exec -s read-only` fallback): mandatory at AG1 exit and AG7 entry; optional elsewhere when a gate turns out riskier than expected.
11. **Web-side commits stay in the web repo** and follow web rules (one finding = one commit; auto-deploys). This program's web touches are enumerated: analytics Worker **auth-branch extension (`android` as an HMAC-verified source)** + the `EVENT_KEYS` 2-event fix (two commits, before AG3), feedback Worker **auth-branch extension + first-class `android` source rendering** (before AG4), overlay validator generalization + runbook update + empty-delta overlay publish (at AG7), landing/Play-badge parity (owner-pick, post-release). Each consuming gate's entry check verifies its prerequisite is LIVE (curl probe), not just committed.
12. Wrap each session per Trigger A; regenerate NEXT-SESSION with the current gate + next entry point.

Sequencing is strict AG1→AG8 with two defined exceptions: (a) the two prerequisite web commits may land in any session before their consuming gate; (b) **the optional owner-triggered "early-clock" split AG7a** — after AG4 exit, the owner may pull AG7's Console/track-setup half forward (**full precondition list, none skippable:** Play account live, listing draft, data safety, content rating, keystore + CI secrets, first manual AAB upload, internal smoke, closed track opened, Discord tester post) so the 14-day clock runs while AG5/AG6 complete; the closed track then gets a WIP-but-usable build, updated at each later gate exit. Without that trigger, AG7 runs whole after AG6 and **the 14-day window is post-AG7 dead time** — filled only by AG8 prep (questionnaire/listing polish), monitoring touchpoints, and non-Android queue work; no parallelism is claimed. **Honest envelope: 12–18 sessions** (gate-sum 11–15; 10–14 optimistic) **plus Play calendar time** (14-day closed test + 7–14-day first review ⇒ 3–5+ weeks wall-clock from closed-test start to live).

## Gate details

### AG0 — Owner ratification + account kickoff (the only pre-code human step)
Owner reads spec §2 + §4 and replies GO or overrides specific decisions (each independently overridable; AD0 scope bar, AD2 applicationId, and the AD1 prototype-push are the three most consequential). Owner actions: (1) **create the Play Console personal account** ($25, ID verification, 2SV, DSA non-trader declaration — the calendar long-pole's first domino; audit-delta §4); (2) **push the mac-mini's 4 unpushed prototype commits** (or write them off explicitly — relaxing the Phase-1 no-push boundary is the owner's call); (3) confirm/deny an Android test device. **The ledger `docs/planning/ANDROID-V1-GATE-LEDGER.md` already exists with a PENDING-GO ratification row (created by the 2026-07-08 planning session to kill the ledger/AG1 chicken-and-egg); whichever session first sees the owner's GO flips that row to GO, quoting the owner's message + date.** Upon recorded GO, one commit updates the top-level `Projects/CLAUDE.md` **and byte-identical `AGENTS.md`** `3dpa-android` row: no-push boundary relaxed per ratified AD1, project ACTIVE, gate-session pushes are protocol step 8's job (repo stays outside `push-children`). **Exit:** GO (or amended decisions) recorded in the ledger + the protocol-row commit.

### AG1 — Foundation (the architecture gate, 2 sessions budgeted)
- **Entry: the AD5 runtime spike (≤ half session, ledger-recorded GO before anything else):** vendor `quickjs-wrapper`; load `engine.js` + `workshop-tuning.js` + `workshop-tuning-rules.js` + `state-codec.js` in one context on the API-36 emulator; run ≥5 golden-snapshot states through `resolveProfile`/`getWarnings`/`getFilters` and value-compare against the web-committed snapshot; verify exception stacks reach Kotlin, 16 KB alignment, real minSdk floor, init latency (<150 ms budget), locale loading via the fetch polyfill. Failure → fallback chain per AD5 (androidx.javascriptengine adapter → re-research), each step ledger-recorded. **Fallback adoption is NOT a silent swap:** androidx.javascriptengine breaks AD4's load-bearing shape (sync single-thread bridge, sync host-function polyfills, plausibly the init budget) — adopting it = an AD4/AD5 spec amendment + hostile re-review of the revised bridge design + an envelope hit (+1–2 sessions), recorded as a spec-change row in the ledger.
- Scaffold: tag prototype `webview-prototype-final`; remove the WebView shell + stale snapshot from the working tree; Gradle Kotlin DSL + version catalog, `:app` module, package `dk.mragile.printassistant`, minSdk 29 / targetSdk 36; port the prototype's `build-debug-apk.sh` intent to Gradle tasks + keep an adb smoke script.
- Asset-mirror pipeline: `scripts/sync-web-assets.sh` (checksums + source-SHA ledger entry — generalizes the prototype's SNAPSHOTS.md idea) + a byte-diff test that fails when bundled assets ≠ recorded web SHA contents.
- `enginebridge/`: `EngineCore` (context lifecycle on a single-thread dispatcher, typed exceptions with JS stacks, fetch/localStorage/console polyfills, deferred-based readiness — no sleep-polling) + `EngineAPI` facades (Profile, Catalog, Troubleshooter, Tuning, Export, I18n).
- **Contract-fixture floor (spec §3.1):** golden mirrors (full web golden-state set through the bridge), codec round-trip + unknown-id degradation, workshop envelope byte-compat (web-generated fixtures; PRETTY 2-space Kotlin emitter), op-log semantics fixtures. All instrumented; all green before AG2 opens. **Sequencing made explicit (Codex CX-4): AG1 builds the CONTRACT layer of `WorkshopStore` (ordered emitter + op-log write APIs + adapter surface) to fixture level only — no UI, no import/export flows; AG4 builds the Workshop UI + backup import/export on top and re-verifies against freshly regenerated web fixtures (AD5a).**
- `designsystem/`: color/type/spacing/radius/motion/haptic tokens (brand per AD7), bundled Syne + DM Mono.
- CI: build + JVM tests + emulator instrumented tests per push (`reactivecircus/android-emulator-runner`); flakiness fallback per spec AD10 recorded if hit.
- **Perf baselines** (cold launch to interactive, engine-init, representative `resolveProfile` bridge call on the pinned AVD) → committed `docs/planning/android-v1/perf-baseline.md` (web repo); AG6 regresses against this.
- **Exit:** fixtures + suite green, hostile review, **Codex cross-model review of the architecture**, ledger tick, pushed.

### AG3 — Output + power tools

Per `feedback_committed_is_not_deployed`, the AG3 micro-plan verifies both are LIVE at gate entry, before any Android code depending on the Worker — with event-specific props in the probe (`troubleshoot_used {symptom}`), since a bare-event curl would pass while dimensions still drop; instrumented tests never hit prod**). **Exit:** standard + packet; RED-first tests mirror web walkthrough invariants where applicable.

### AG4 — Workshop, journal, troubleshooter, feedback
**Entry: AD5a re-verify (≤ 1 hour, ledger-recorded):** adapter surface + Kotlin store write-APIs fixture-checked against fresh web-generated op-log fixtures (the architecture is already decided — this verifies bytes, not direction). Then: Workshop (saved profiles CRUD, JSON backup export/import — **import must accept real web AND iOS backup files, fixture-pinned**), journal UI (worked/failed + symptom tags + note, 500-char cap, troubleshooter deep-link — web W2 semantics), troubleshooter surface (`getSymptoms`/`getTroubleshootingTips` with materialGroup context) + `troubleshoot_used` event, feedback form with contextual prefill-as-argument (iOS S1 lesson) → Worker (**prerequisite: the web auth-branch + `android`-source commit — verified LIVE at gate entry with an `X-App-Source: android` probe, before any Android code depending on the Worker; same committed≠deployed discipline as AG3**), session persistence (codec-FIELDS snapshot + unknown-id degradation), **overlay consumer (AD8): remote fetch + `payload_sha256` verification + `min_app_version` gating + cache + override-merge-by-id, fixture-tested against a local same-shaped fixture (the live artifact doesn't exist until AG7 — its end-to-end check happens there).** **Exit:** standard + packet for the new screens. This is the heaviest gate (2–3 sessions); the recorded split option is workshop+persistence+overlay first, journal+troubleshooter+feedback second.

### AG5 — Native Mine loop + share (the payoff gate)

### AG6 — Polish, a11y, i18n audit

### AG7 — Release train part 1 (owner-interleaved)
Entry: **Codex cross-model review of the full scaffold→v1 diff**; triage CRITICAL→LOW, one finding = one commit. Then: assistant drafts store listing (title/short/full description, feature graphic + screenshots from the harness, EN+DA), data-safety form mapping (audit-delta §4.8) and content rating answers — **owner submits in Console**; owner generates the upload keystore (assistant-scripted bare commands) + adds CI secrets (base64 keystore, passwords, GCP service-account JSON — service account granted release permission in Console); **owner performs the first manual AAB upload** (Play quirk); internal-track smoke (owner + emulator matrix); **closed testing opens** + owner posts the assistant-drafted tester call in Discord (≥12 testers, 14-day continuous engagement — the clock starts here); web-side: **first make the existing iOS overlay validator green (currently RED — missing 1.0.7 baseline; Codex CX-2) or decouple the Android validator path from iOS baseline freshness**, then overlay validator platform-generalization + runbook extension + publish an (initially empty-delta) `android-printer-overlay-v1.json` + **end-to-end overlay-consumer check against the live artifact** (the AG4-built consumer fetches, sha256-verifies, and merges it). **Exit:** closed test running, ledger records the clock-start date **+ the mid-window owner checkpoints (owner checks Console engagement stats at ~day 5 and ~day 10 — the assistant cannot see the Console; assistant drafts Discord nudges if engagement lags; insufficient engagement is the top 2026 rejection reason and must not be discovered on day 14). Window sessions (if any) do AG8 prep (questionnaire/listing polish), monitoring, or non-Android queue work.**

### AG8 — Release train part 2 (after the 14-day clock)
Verify closed-test engagement ≥12/14d → owner files the production-access application (questionnaire; assistant drafts answers) → production submission (first review 7–14 days) → staged rollout (20% → 100% owner-paced) → **LIVE**. Post-live flips, each its own commit: Android `main` adopts the release-gating rule (mirrors track state); `printers.json` leaves the sync sweep (baseline frozen; overlay model active per AD8); `3dpa-context.md` gains the Android surface row + drops "no plans" residue; ROADMAP + memory hot-cache updated; the 12-tester cohort thanked/retained for future closed tracks. **Rollback:** pre-production = tracks are independent, nothing public; post-release = staged-rollout halt + `versionCode`-bumped fix release (Play has no binary rollback — halt-and-fix is the mechanism, worth stating honestly).

## Risk register
