## 3. Release scope

### Included

1. **#2 — persistent saved-state guard.** A loaded or just-saved Workshop
   profile cannot be saved again until its normalized configuration changes.
2. **#3 — permanent Discord invite.** iOS uses
   `https://discord.gg/4KmcHrPkcS`.
3. **#4 — visible, honest export UX.** Workshop backup export/import is visible;
   users choose all profiles or one named profile; Output exposes a labelled
   action that distinguishes native Bambu JSON from Orca/Prusa text sharing.
4. **#5 — end-to-end feature analytics.** iOS reports successful Workshop and
   export interactions; the web Worker accepts them and the owner dashboard
   displays them without expanding the privacy boundary.
5. **1.0.7 release train.** The existing K2 SE bundled mirror rides the build;
   full tests, review, TestFlight dispatch, owner QA, and only then App Store
   submission preparation.

### Excluded

- Native OrcaSlicer JSON and PrusaSlicer INI serializers. The UI must state the
  current behavior honestly; serializer work remains a fixture/import-tested
  export phase.
- Android implementation, iOS v2 rebuild work, selection-event analytics, and
  unrelated refactoring.
- Merging the July 9 web branch wholesale. Only the current-main-compatible #5
  delta is implemented.
- Closing #4 as fully resolved while native Orca/Prusa file export remains open.
- App Store submission before owner acceptance of the new TestFlight build.

The final release claim is intentionally split: Orca/Prusa **formatted-text
sharing** may be called live only after exact-build device verification and a
public 1.0.7 availability check. Native Orca JSON and Prusa INI export remain
not live and must never be inferred from that text-share result.

## 4. Architecture

### 4.1 Saved-state guard (#2)

`WorkshopStore` owns the saved-profile lifecycle and therefore owns the active
saved snapshot. It will expose a normalized fingerprint contract instead of
letting `OutputView` keep a time-based UI flag.

The fingerprint is generated from `AppStateWebCodec.webDict(from:)`, recursively
normalizes dictionaries, sorts string arrays whose order is not semantic, and
serializes with sorted keys. `restore` records the post-codec state fingerprint;
successful `save` records the new profile id and fingerprint; `remove` clears
the active marker when that profile is deleted.

`OutputView` derives its Save state from
`WorkshopStore.isCurrentStateSaved(_:)`. It shows the filled-star/Saved state
and disables Save while the fingerprint matches. Any state change that alters
the fingerprint re-enables Save; changing back to the saved state disables it
again. A small view-local revision counter only forces reevaluation after a
successful store mutation—it is not the source of truth.

### 4.2 Discord link (#3)

`AppConstants.discordURL` is replaced with the permanent owner-supplied URL and
pinned by a unit test. No feedback webhook or Worker behavior changes.

### 4.3 Workshop backup selection (#4a)

The Workshop view gains a visible action row with labelled Export and Import
buttons. The overflow-only entry is removed. Export behavior is:

- zero profiles: Export disabled; Import remains available;
- one profile: export that full Workshop envelope directly;
- multiple profiles: present a sheet with “All profiles (N)” and each named
  profile;
- all-profile export: byte-identical web-compatible full envelope including
  additive extras (the existing ordered emitter and round-trip tests pin this)
  such as tuning;
- single-profile export: one profile in the same v1 envelope, with global
  tuning/additive extras omitted so unrelated Workshop data is not disclosed.

The single-profile filename uses a bounded safe slug. Import success names the
number imported. Analytics fires only after a successful file write/import.

### 4.4 Output export discoverability (#4b)

The dormant `exportMenu` is replaced by a visible labelled row below the output
mode control, avoiding an overloaded navigation bar.

- Bambu Studio: **Export for Bambu Studio** invokes the existing native
  process+filament JSON share path.
- OrcaSlicer: **Share settings for OrcaSlicer** invokes existing formatted-text
  sharing.
- PrusaSlicer: **Share settings for PrusaSlicer** invokes existing formatted-text
  sharing.

The app never labels text sharing as a native slicer file. There is no manual
slicer picker: the engine-resolved slicer for the selected printer determines
the action, preventing invalid printer/slicer combinations.

### 4.5 Analytics (#5)

The current 20-blob schema remains unchanged. `blob19` stays the shared
`eventDetail` field. The current property name `type` is reused instead of the
stale branch’s `exportType`/`exportScope` names.

New accepted events:

| Event | Fires after | `type` detail |
|---|---|---|
| `workshop_saved` | profile write succeeds | empty |
| `workshop_loaded` | restore succeeds and output navigation is initiated | empty |
| `workshop_exported` | backup file write succeeds | `all` or `single` |
| `workshop_imported` | import validates and writes | empty |

Existing `export_clicked` is emitted from iOS when the user invokes the visible
Output action, with `type=bambu_bundle` or `text_share` plus printer/material/
nozzle ids. It measures the same click/intent boundary already used by web; file
write failures remain visible through the existing error path.

The dashboard adds one combined Feature Usage query/panel covering
`export_clicked`, the four Workshop events, and `troubleshoot_used`. Event
payloads remain aggregate-only: no user/session/device identifiers, profile
output, custom profile name, free text, file contents, or cross-session history.
Existing App Privacy declarations remain accurate.

### 4.6 Release mechanics

- Keep `MARKETING_VERSION` at **1.0.7**.
- Do not edit `CURRENT_PROJECT_VERSION`; Fastlane sets a unique
  `YYYYMMDDHHmm` build number during the workflow.
- Keep iOS commits local until the entire train is TestFlight-ready.
- Push iOS `main` once after all fixes, review patches, full tests, shared-file
  identity checks, and release notes are complete.
- Dispatch `.github/workflows/testflight.yml` manually against `main` and monitor
  through successful upload and App Store Connect processing.
- Owner tests the exact newest 1.0.7 build. Public submission is a separate gate.

## 5. Commit and review policy

- One owner-reported finding or accepted review finding per commit.
- Web analytics contract and dashboard changes stay in their own web commits.
- Planning artifacts are committed before implementation.
- A completion claim requires clean worktrees in every touched repo. Intentional
  local commits/ahead state must be explicit while gated; no uncommitted dirty
  state may remain at wrap-up or release completion.
- Plan gate: independent Claude review through `bridge --mode claude-only`.
- Implementation final gate: bounded cross-model review after full local QA.
- A reviewer verdict is evidence, not permission to push or submit.

## 6. Verification and acceptance

### Automated

- Web analytics Worker/query tests: all green, including new Workshop events and
  Feature Usage SQL.
- iOS focused tests for saved-state fingerprinting, selective backup export,
  analytics payloads, and the permanent Discord URL.
- Full iOS XCTest suite and relevant UI tests.
- UI screenshot walkthrough showing the visible Workshop and Output actions.
- `cmp` for web/iOS `engine.js` and the full shared `data/` tree.
- Web `validate-data`, overlay validator, walkthrough harness, and
  `git diff --check` because the train includes the pre-existing K2 SE bundled
  data commit.

### Owner TestFlight QA

1. Load a Workshop profile: Save is disabled/Saved.
2. Change one setting: Save re-enables; revert it: Saved returns.
3. Open Discord from Home: permanent invite resolves.
4. Workshop actions are visible without the overflow menu.
5. Multiple profiles offer All versus one named profile; both files import.
6. Full backup preserves Mine tuning round-trip.
7. Output action says Bambu Studio, OrcaSlicer, or PrusaSlicer and performs the
   documented native-file/text-share behavior.
8. K2 SE is present in the bundled picker while live overlay behavior remains
   healthy.
9. New feature events appear in the analytics dashboard after ingestion delay.

## 7. Stop and rollback rules

- If baseline tests are red before implementation, stop and diagnose.
- If shared engine/data identity breaks unexpectedly, stop; do not patch iOS
  independently.
- If web analytics deployment rejects a real signed iOS smoke payload, do not
  dispatch TestFlight.
- If GitHub Actions fails, diagnose and rerun only after the cause is corrected;
  do not stack dispatches.
- If owner TestFlight QA fails, keep 1.0.7 out of App Review and fix through the
  same TDD/review discipline.
- Web rollback: revert the narrow analytics commits and verify production.
- iOS rollback before public release: do not select/submit the failed build;
  upload a corrected newer 1.0.7 build.
