# iOS 1.1.0 Notification Release Design

**Status:** Draft for hostile cross-model review; implementation not started

**Date:** 2026-07-18

**Owner-approved release composition:** notifications + issues
[#2](https://github.com/mustiodk/3dprintassistant/issues/2)–
[#5](https://github.com/mustiodk/3dprintassistant/issues/5) + visible native
Bambu/Orca/Prusa export + the nine gated iOS data/export commits already local

**Supersedes:**
[`2026-07-11-ios-1.0.7-issue-fix-release-design.md`](2026-07-11-ios-1.0.7-issue-fix-release-design.md)
and its 1.0.7 implementation plan. Neither superseded document may be executed
without first being replaced by a 1.1.0 implementation plan derived from this
reviewed design. It also retires the unstarted **1.0.8** version reservation in
the tip-jar plan; the tip-jar product decisions remain parked, but that feature
must receive a version greater than 1.1.0 at its future owner gate.

## 1. Goal and audience

Ship one polished **iOS 1.1.0** release that lets users explicitly opt in to
occasional notifications about newly supported printers and meaningful product
updates, completes the open issue-fix/export work, and leaves App Store copy,
privacy declarations, and owner operations ready for release.

The audience is the existing iOS user base. Push is an optional convenience,
never a requirement for profiles, Workshop, export, or any offline feature.

- Public App Store version: **1.0.4**.
- `3dprintassistant-ios/project.yml` currently says
  `MARKETING_VERSION: "1.0.7"`; 1.1.0 has not been applied.
- Web `main` is clean and aligned with `origin/main`.

## 3. Release composition

### 3.1 Included

1. Native APNs notifications with explicit topic consent, in-app opt-out,
   first-party token storage, owner-only canary/broadcast tooling, and delivery
   observability.
2. Two notification topics only:
   - **New printers** — newly supported printer models.
   - **App updates** — meaningful new or changed app functionality.
3. #2 persistent Workshop saved-state guard.
4. #3 permanent Discord invite `https://discord.gg/4KmcHrPkcS`.
5. #4 visible Workshop backup/import and visible Output export.
6. #5 aggregate Workshop/export analytics and owner dashboard coverage.
7. Native file output for the engine-selected slicer:
   - Bambu Studio process + filament JSON;
   - OrcaSlicer process + filament JSON;
   - PrusaSlicer config-bundle INI.
8. The nine already-local iOS commits, subject to full current-main validation.
9. Version 1.1.0, updated privacy policy/labels, release notes, promotional
   text, review notes, screenshots if the visible release UI changes materially,
   TestFlight acceptance, and App Store submission checklist.

### 3.2 Excluded

- Tip jar, donations, subscriptions, purchases, accounts, or entitlements.
- Android notifications or web push.
- A notification inbox, notification history screen, campaign CMS, scheduler,
  segmentation, personalized copy, A/B testing, or engagement tracking.
- Promotions, offers, fundraising, ratings requests, or sensitive/personal data
  in a push payload.
- Background/silent pushes. Version 1 uses user-visible APNs alert pushes only.
- Downloaded executable code or remotely changed engine logic. Remote printer
  updates remain strictly validated additive data.
- Pushing iOS commits, dispatching TestFlight, or submitting to App Review
  before all release gates pass and the owner explicitly approves the ship step.

### 3.3 Version-train correction

This release **replaces**, rather than follows, the unstarted 1.0.7 issue-fix
and 1.0.8 tip-jar trains. App Store version numbers must move forward:

- notification/fixes/exports/printers: **1.1.0** (this design);
- tip jar: still separate and parked; recommended next feature version is
  **1.2.0**, to be ratified when that work is resumed—never 1.0.8;
- My 3DPA/accounts: remains parked and receives a new later version only at its
  own owner gate; every existing “iOS 1.1.0” reservation in those plans is stale.

No parked feature is pulled into this release by renumbering it.

## 4. Decision and alternatives

### 4.1 Decision: native APNs + first-party Cloudflare provider

Use `UserNotifications` and UIKit registration in the app. Extend the existing
Cloudflare Worker as the provider, with a new EU-jurisdiction D1 database for
consented device routes and a Queue + dead-letter queue for bounded APNs
delivery. Use Apple's token-based APNs authentication with a dedicated `.p8`
key stored only as Cloudflare secrets.

This keeps the app free of another tracking-capable SDK, reuses the existing
first-party service boundary, and makes consent, retention, delivery, and
rollback inspectable by the owner.

### 4.2 Rejected alternatives

| Alternative | Rejection reason |
|---|---|
| OneSignal or Firebase Messaging SDK | Adds a third-party data processor/SDK and a broader privacy/vendor surface for a two-topic feature. |
| Local notifications only | Cannot tell an inactive app that a printer or release was added. |
| In-app “What's new” only | Useful later, but does not satisfy the owner's request to notify users outside the app. |
| Direct fan-out in one admin HTTP request | Couples owner UX to APNs latency and Worker request limits; partial failure/retry becomes unsafe. |
| APNs broadcast channels | Adds newer channel-management machinery without solving the required per-install explicit topic consent and opt-out audit trail. |

## 5. User experience and consent

### 5.1 Entry point

Home gains one quiet, secondary row below the existing core actions:

- before consent: **Product updates** · `Off`;
- after consent: **Product updates** · selected topic summary;
- when system permission is denied: **Product updates** · `Blocked in Settings`.

The row opens a native sheet. The app never opens either its own sheet or the
system notification prompt automatically on launch.

### 5.2 Pre-permission sheet

Locked copy:

> **Stay in the loop**
>
> Choose the occasional updates you want from 3D Print Assistant. No promotions,
> no daily noise, and you can turn them off here at any time.

Both topic toggles start **off**. The primary **Enable selected notifications**
button remains disabled until the user turns on at least one topic:

- **New printers** — Know when another printer is ready to use in the
  configurator.
- **App updates** — Hear about meaningful new tools and improvements.

Only that button may trigger Apple's `.alert` + `.sound` authorization prompt.
No badge permission is requested, and no time-sensitive or critical entitlement
is used.

This two-step consent is deliberate: Apple treats product/marketing pushes as
requiring explicit in-app consent language plus an in-app opt-out. The settings
remain accessible regardless of Apple's current authorization state.

### 5.3 Preference behavior

- If Apple authorization succeeds, the app registers with APNs and upserts only
  the selected topics after receiving the token.
- If authorization is denied, no token is uploaded. The sheet explains that
  notification permission is off and offers **Open iOS Settings**.
- Toggling one topic off immediately updates the server registration.
- Toggling both topics off calls `unregisterForRemoteNotifications()` and marks
  the last token as **pending deletion** locally until the server confirms the
  idempotent unregister. Network failure retries on foreground/launch; the app
  must not discard its only deletion handle before a 2xx response.
- On every foreground/launch after opt-in, the app rechecks Apple's settings.
  If permission was revoked externally, it deletes the last known server token
  when possible and moves the UI to `Blocked in Settings`.
- After opt-in, the sheet footer shows a one-way, 16-hex-character
  **Notification ID** derived from the server token hash. It is used only to
  select the owner's canary device or diagnose a support request. Ambiguous
  prefixes are rejected rather than guessed. It is not an account, analytics
  identifier, or payload field.

### 5.4 Cadence and content rules

- Maximum **one public broadcast in any rolling seven-day period**. Multiple
  printers are grouped into one message.
- Canary sends do not count against that limit.
- Every campaign targets exactly one consented topic.
- Copy must be factual, short, and connected to functionality already available
  to the target build/catalog.
- Never send promotions, tip requests, purchase prompts, ratings requests,
  personal content, or sensitive/confidential information.
- Notifications use normal alert delivery; never `time-sensitive` or `critical`.
- No per-user engagement/open receipt is collected. APNs acceptance is recorded
  as provider delivery, not falsely described as device receipt or user open.

## 6. Notification interaction contract

### 6.1 Payload v1

```json
{
  "aps": {
    "alert": { "title": "A new printer is ready", "body": "Snapmaker U1 is now supported." },
    "sound": "default",
    "thread-id": "3dpa-new-printers"
  },
  "schema_version": 1,
  "announcement_id": "printer-snapmaker-u1-20260718",
  "kind": "new_printer",
  "brand_id": "snapmaker",
  "printer_id": "snapmaker_u1"
}
```

Allowed `kind` values and extra fields:

| Kind | Required custom fields | Tap result |
|---|---|---|
| `new_printer` | `brand_id`, `printer_id` | Once the engine is ready, select the brand and open the printer picker with the model highlighted. |
| `app_update` | `release_version` | Open the exact first-party App Store listing for 3D Print Assistant. |

The app ignores unknown schema versions/kinds and opens Home safely. It never
opens an arbitrary URL supplied in a payload. Title/body are display data only;
they never alter app behavior.

Provider headers are pinned to the production bundle topic, alert push type,
normal priority, a bounded expiration, and
`apns-collapse-id=<announcement_id>` so safe retries do not create repeat alerts.
The complete payload must remain below APNs' documented size limit.

### 6.2 Cold/warm launch routing

`NotificationCoordinator` owns one pending destination. The app delegate passes
notification-response payloads to it; `ContentView` consumes a valid destination
only after the engine is ready. Warm-app taps and cold-launch callback races are
both test cases, not timing assumptions.

For `new_printer`, launch changes from “cache now, use next launch” to:

1. load the bundled + last-known-good cached overlay immediately;
2. call the existing validated `PrinterCatalogProvider.prepareForLaunch()`
   before taking the shared Swift/JSCore catalog snapshot;
3. continue from cache/bundle on timeout or validation failure;
4. initialize `DataService` and `EngineService` from the same immutable snapshot;
5. route only if the requested brand/printer exists in that snapshot.

The current network request has a two-second hard timeout. Added launch delay is
measured during implementation; a healthy-network p95 regression over 750 ms is
a release blocker and must be solved without weakening validation or making
Swift/JSCore catalogs diverge.

If the target printer is still absent, the app remains on Home and shows:

> **Printer update not ready** — We couldn't load this printer yet. Check your
> connection and try again.

No force-restart instruction, crash, invalid selection, or partially refreshed
catalog is acceptable.

## 7. iOS architecture

### 7.1 Components

| Component | Responsibility |
|---|---|
| `AppNotificationDelegate` | `UIApplicationDelegate` + `UNUserNotificationCenterDelegate`; APNs callbacks, foreground presentation, response forwarding. Wired through `@UIApplicationDelegateAdaptor`. |
| `NotificationService` | Main-actor observable service for authorization state, local topic choices, APNs registration lifecycle, signed register/unregister calls, and user-facing errors. |
| `NotificationCoordinator` | Validates payload v1, stores one pending route, and coordinates engine-ready navigation without owning permission/token state. |
| `ProductUpdatesView` | Pre-permission copy, topic toggles, blocked-state Settings link, opt-out, and Notification ID. |
| `AppRouter` additions | Typed printer notification target plus safe Home fallback. No raw URL route. |

Topic choices persist locally in `UserDefaults`. The last APNs token and pending
unregister state are stored with iOS data protection solely so an external
permission revocation or failed deletion can be retried; they are cleared only
after confirmed unregister. Token callbacks are forwarded every time APNs
supplies a token—token length or stability is never hard-coded.

### 7.2 Entitlements and environments

- Add Push Notifications capability through xcodegen-managed project settings
  and a source-controlled entitlement file.
- Do not add Background Modes; version 1 sends no silent/background pushes.
- Debug/local device registration uses APNs development; TestFlight/App Store
  use production. The server row records the environment and chooses the
  matching APNs endpoint.
- Archive QA must inspect the signed app entitlement and prove production APNs
  before TestFlight dispatch.

### 7.3 Failure behavior

- Permission, token registration, or server failures never block app use.
- Registration failure leaves selected local topics intact and shows a retry
  state; it never claims notifications are active.
- Register/unregister requests are idempotent. Network retries cannot create
  duplicate device rows.
- Foreground notifications present through the normal system alert/sound policy.
- No APNs token, notification hash, authorization header, or push secret may be
  recorded in Sentry, analytics, console logs, review artifacts, or CLI output.

## 8. Provider architecture

### 8.1 Routes and trust boundaries

Add explicit method/path dispatch in `worker.js`:

| Route | Auth | Purpose |
|---|---|---|
| `POST /api/push/register` | Dedicated iOS push-registration timestamped HMAC | Validate token/topics/environment; encrypt token; idempotently upsert consent. |
| `POST /api/push/unregister` | Dedicated iOS push-registration timestamped HMAC | Delete the matching token row and outstanding device delivery rows. |
| `POST /api/push/admin/campaigns` | Dedicated constant-time bearer token | Validate a canary or public campaign, enforce idempotency/cadence, create campaign, enqueue fan-out. |
| `GET /api/push/admin/campaigns/:id` | Dedicated bearer token | Return aggregate counts/status only; never return tokens or ciphertext. |

`IOS_PUSH_REGISTRATION_SECRET` is separate from the existing feedback/analytics
secret so push abuse or rotation does not couple to Android or older feature
clients. It is still only abuse friction, not user/device authentication—it can
be extracted from a binary. Registration endpoints therefore also have strict
body/schema limits and a Workers rate-limit binding keyed by source IP without
persisting the IP. The admin token is a separate Cloudflare secret and is never
embedded in either app.

### 8.2 Storage

Create `3dpa-push-production` with
`npx wrangler@latest d1 create 3dpa-push-production --jurisdiction=eu`.
Jurisdiction is creation-time only; a Western Europe location hint is not an EU
residency substitute.

Checked-in numbered migrations create:

```text
push_devices
  environment + token_hash PK      development | production + SHA-256 token
  token_ciphertext, token_iv       AES-GCM; raw token never stored/logged
  encryption_key_version
  topics                          bit mask: new_printers | app_updates
  app_version, build_number
  created_at, updated_at, last_apns_success_at

push_campaigns
  campaign_id PK
  topic, kind, title, body
  brand_id?, printer_id?, release_version?
  audience_mode                   canary | public
  status                          queued | sending | complete | partial | failed
  created_at, started_at, completed_at
  accepted_count, invalid_count, failed_count

push_deliveries
  campaign_id + token_hash PK
  status                          pending | apns_accepted | consent_removed | invalid | retryable | failed
  attempts, last_apns_status, updated_at
```

Raw APNs tokens are encrypted with AES-GCM using a versioned Worker secret.
`token_hash` is a one-way high-entropy lookup/support identifier. Neither value
is sent to Analytics Engine.

Retention:

- opt-out/unregister: device row and its device-specific delivery rows deleted
  immediately;
- APNs `410`, `BadDeviceToken`, or `DeviceTokenNotForTopic`: device row deleted
  immediately;
- delivery rows: deleted after 30 days by a daily Wrangler Cron maintenance
  handler with focused retention tests;
- aggregate campaign row: retained for operational history, with no device id;
- stale device rows with no APNs success or registration refresh for 180 days:
  deleted.

The privacy policy says exactly this and names Cloudflare and Apple as service
providers. No IP, user-agent, locale, user/account id, notification-open event,
or generated profile data is persisted.

### 8.3 Queued fan-out and idempotency

`POST /api/push/admin/campaigns` is dry-run unless the request contains the
explicit send confirmation for the same `campaign_id` and preview digest. A
duplicate public `campaign_id` is rejected. Public creation is rejected if any
non-cancelled public campaign was created inside the rolling seven-day window;
a failed/partial attempt cannot silently bypass cadence with a new id.

The producer enqueues a campaign/page cursor, not raw tokens. Queue consumers:

1. page eligible token hashes from D1 in stable order;
2. skip a delivery already marked `apns_accepted`, `consent_removed`, or
   `invalid`;
3. if its `push_devices` row disappeared through opt-out/invalidation, mark the
   delivery `consent_removed` and continue without retrying or sending to DLQ;
4. decrypt only the current batch's tokens in memory;
5. reuse a valid APNs ES256 provider JWT, refreshing within Apple's required
   window;
6. send with bounded concurrency;
7. persist each APNs response before advancing the cursor;
8. retry only `429`/`5xx`/transport failures with delay;
9. delete invalid tokens on terminal APNs token errors;
10. place exhausted retryable work on the configured dead-letter queue.

APNs 2xx means **accepted by APNs**, not delivered to a device. Campaign status
and CLI wording use that exact distinction. `apns-collapse-id` plus the delivery
ledger makes at-least-once Queue processing user-safe.

### 8.4 Owner CLI and runbook

`scripts/send-ios-push.mjs` is the only supported send surface:

1. validate copy/payload locally;
2. preview matching consented device count and a SHA-256 preview digest;
3. send to one exact Notification ID canary;
4. owner confirms receipt and tap behavior on the target TestFlight build;
5. repeat preview, then type the campaign id to confirm one public send;
6. poll aggregate APNs accepted/invalid/retry/failed counts;
7. stop on any unexpected partial/DLQ state and follow the runbook.

The CLI reads its admin token from a local ignored environment file or Keychain,
never a command-line argument. It prints no token, ciphertext, secret, or full
Notification ID. A runbook covers APNs-key rotation, encryption-key rotation,
DLQ inspection/replay, D1 migration order, canary send, opt-out verification,
and kill/rollback steps.

## 9. Existing release work folded into 1.1.0

### 9.1 Issues #2–#5

The normative saved-state, Discord, Workshop backup, and analytics behavior is
the reviewed 1.0.7 design §§4.1–4.5, with these 1.1.0 amendments:

- version and ship gates are 1.1.0;
- #4 Output labels are native file exports for all three slicers, not text
  sharing for Orca/Prusa;

- #4 may close only after all three exact TestFlight outputs pass real desktop
  slicer imports.

### 9.2 Native export UI

A visible labelled action row sits below the Simple/Advanced control. The
engine-resolved slicer determines exactly one action:

- **Export for Bambu Studio** → two JSON files;
- **Export for OrcaSlicer** → two JSON files;
- **Export for PrusaSlicer** → one INI config bundle.

There is no manual slicer override. File creation is asynchronous, errors remain
visible/reportable, analytics fires at the same user-intent boundary as web,
and the share sheet opens only after every expected file writes successfully.

### 9.3 Data/logic impact evaluation

- **iOS:** bundled printer data and engine serializers are already local but
  require current-main regression proof and UI exposure.
- **Web:** no engine/data behavior change is needed for these mirrors. The web
  Worker gains notification routes/storage/queue plus the narrow #5 event/query
  work from the superseded design.
- **Remote overlay:** remains additive data-only. The pre-engine validated
  refresh makes newly announced printers usable on the notification-open launch
  when the network succeeds.
- **Android:** no consumer or implementation change. The payload schema is
  versioned so a later Android design can coexist without changing iOS v1.
- **UI/UX:** Home gains one secondary row/sheet; Output gains one visible export
  row; Workshop gains visible backup/import actions. No navigation redesign.

## 10. Privacy, security, and App Store declarations

### 10.1 App Privacy change

The APNs token is a persistent device-level route retained by the first-party
provider. The conservative App Store Connect declaration is therefore:

| Data type | Collected | Linked to identity | Tracking | Purpose |
|---|---|---|---|---|
| Identifiers → Device ID | Yes, only after opt-in | No | No | App Functionality |

Existing Diagnostics and Usage Data declarations remain. The final ASC screen
must be owner-verified against Apple's then-current questionnaire; this design
must not reuse the old “Device ID: No” answer.

### 10.2 Security controls

- Dedicated APNs signing key; do not reuse an App Store Connect API key.
- `.p8`, APNs key id/team id, admin bearer, and encryption key are Cloudflare
  secrets. Secret values never enter git, docs, screenshots, or Bridge prompts.
- AES-GCM at rest, HTTPS in transit, EU D1 jurisdiction, input allowlists,
  bounded bodies, native registration rate limit, constant-time admin auth,
  idempotent campaign ids, delivery retry caps, and DLQ.
- Notification payloads contain public product data only.
- No token enumeration route. Admin reads aggregate results only; canary lookup
  requires an exact Notification ID.
- `.assetsignore` already excludes `scripts/**` and `docs/**`; implementation
  must add the new migration directory plus every local secret/config pattern
  before the first deploy. Public probes must prove those paths and Worker
  source/config are not served by the static asset binding.

## 12. Gates and verification

### G0 — owner configuration gate

Owner actions that cannot be inferred or fabricated:

1. enable Push Notifications for the production App ID;
2. create/download a dedicated APNs `.p8` key and provide key id/team id through
   the approved secret-entry path;
3. approve creation of EU D1, Queue, DLQ, rate-limit binding, and Worker secrets;
4. verify the App Store Connect Device ID privacy answer and final privacy text.

No secret is pasted into chat or committed. Implementation may proceed up to a
tested local/mock boundary before G0, but no production push claim may pass it.

### Pre-push automated gates

- TDD-focused Worker route, schema, migration, crypto, campaign, cadence,
  idempotency, APNs response, retry, invalidation, retention, and redaction tests.
- Local D1/Queue integration through the real Wrangler/Miniflare bindings—not
  only hand-written mocks.
- iOS tests for permission states, topic persistence, registration rotation,
  opt-out, payload validation, cold/warm pending routing, missing-printer
  fallback, and no-token logging.
- Existing #2–#5 focused tests plus Bambu/Orca/Prusa file tests.
- Full iOS XCTest and UI walkthrough harness.
- Exact shared-file identity and existing web validate-data/overlay/walkthrough/
  export audit gates.
- Archive entitlement inspection and static-asset exposure probes.
- `git diff --check`, privacy/copy character checks, and clean worktrees.

During local implementation, `MARKETING_VERSION` is changed to **1.1.0** and the
generated Xcode project is refreshed before these final gates. After G0,
development-environment registration/APNs canary, pre-push automation, final
cross-model implementation review, and release-copy QA are green, the owner may
confirm that the complete train is ready to ship to TestFlight. Only then is the
single gated iOS `main` push allowed, followed by the manual TestFlight dispatch.

### Owner TestFlight gates

1. Permission never appears until the owner selects a topic and enables it.
2. Deny, allow, single-topic, two-topic, external Settings revoke, and in-app
   full opt-out all show truthful state.
3. Canary `new_printer` notification arrives, displays correctly, and routes to
   the intended printer or the explicit safe fallback.
4. Canary `app_update` notification opens the exact App Store listing.
5. Opted-out device is absent from a new canary attempt.
6. Bambu JSON, Orca JSON, and Prusa INI generated by the exact build import in
   their real desktop slicers.
7. #2–#5 acceptance from the superseded design passes with native export labels.
8. Bundled printers work offline and remote overlay validation still passes.
9. App Store copy, privacy policy, privacy labels, and screenshots match the
   exact build.

Only after all nine exact-build TestFlight gates pass may that build be
considered for App Review.

## 13. Rollout, rollback, and stop rules

- Worker deploy order: provision EU D1/Queue/DLQ and secrets → apply numbered
  migrations → deploy with `PUSH_REGISTRATION_ENABLED=false` and
  `PUSH_PUBLIC_SEND_ENABLED=false` → enable registration for development/
  TestFlight canaries → prove canary registration/send → keep public broadcast
  disabled until the owner deliberately sends after release. There is no public
  audience before the iOS opt-in UI ships.
- `PUSH_REGISTRATION_ENABLED` never disables unregister. Canary send remains
  possible while `PUSH_PUBLIC_SEND_ENABLED=false`; that flag gates public
  fan-out only.
- If token encryption/decryption, APNs JWT, production entitlement, or canary
  behavior cannot be proven, notification release is **NO-GO**; do not silently
  ship a broken opt-in row.
- If D1/Queue migration or Worker deploy fails, stop and restore the previous
  Worker; never point the app at a partially migrated route.
- If public send becomes unsafe, disable campaign send first. Do not disable
  unregister/opt-out.
- If APNs key compromise is suspected, disable sends, revoke/replace the key,
  and follow the key-rotation runbook before resuming.
- If the exact TestFlight build fails any export import or issue acceptance,
  keep it out of App Review and correct it with the same TDD/review gates.
- Web engine/data changes are not authorized by this design. Any newly discovered
  need there returns to owner scope approval.
- iOS push gate remains closed throughout planning and implementation until the
  complete 1.1.0 train is owner-ready for TestFlight.

## 14. Definition of spec approval

This design becomes implementation-plan-ready only when:

1. the hostile cross-model review has a preserved non-empty transcript;
2. every P0/P1 and accepted P2 finding is dispositioned in the review ledger,
   with one accepted finding per commit;
3. the final self-review finds no placeholders, contradictory version/export/
   privacy claims, or secret-bearing examples;
4. the owner explicitly approves or amends the final design and G0 boundary.

The next artifact after owner approval is a gated, test-driven 1.1.0
implementation plan. This document authorizes no code, infrastructure, push,
TestFlight dispatch, or App Store submission by itself.
