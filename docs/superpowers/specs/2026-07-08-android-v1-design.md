## 1. Owner intent (restated + patched)

> Audience/scope block per `feedback_owner_intent_explicit_in_foundation_docs`.

The owner corrected the 2026-07-08 goal: the target is **an Android app first** (an improved Android version of what the iOS app is), then a macOS app later; the iOS v2 rebuild is PARKED as a program and reused as evidence. This session produces the Android planning bundle (audit-delta → this spec → gated master plan), triple-reviewed; **execution happens in later gated autonomous sessions**, one gate per session, with per-gate micro-plans authored at gate entry.

**Patched interpretation (holes filled — owner ratifies at Gate 0):**
- "Android app" = a **native Android app** (Kotlin + Jetpack Compose) with the same one-engine architecture as web/iOS: `engine.js` + `data/*.json` byte-identical, run in an embedded JS runtime, **never ported to Kotlin**. Not the parked WebView wrapper (audit-delta §5).
- It ships to **Google Play as v1.0.0** under the owner's (to-be-created) personal Play account, free, no accounts, privacy-first — same product stance as web/iOS.
- Quality bar = the iOS v2 §3 pattern: contract fixtures + golden mirrors + TDD RED-first + hostile review per gate + cross-model review at the architecture and release gates. "Quality > speed" stands; MVP-era corner-cutting does not return just because the platform is new.
- macOS is explicitly **second priority, sequenced after Android ships** — companion doc only; nothing in this spec blocks on it.

## 2. Decisions (AD0–AD10) — defaults chosen for autonomy; owner can override any at Gate 0

**AD0 — v1 scope bar: improved-from-day-one (NOT iOS-v1.0.7 parity).** The explicit scope question answered: Android v1 bakes in the iOS-v2 improvement content — native learns-loop (journal → harvest → suggestions → accept → **Mine**), troubleshooter, workshop + byte-compat backup, outbound share, print-time estimator, purge calculator, a11y as a build norm, i18n EN+DA — i.e., **full web feature parity where it makes sense natively**, matching the iOS v2 target surface (its D3), minus iOS-only concerns. *Justification:* (a) the audit names the learns-loop the product's differentiator — shipping a third platform without it re-creates the exact gap that motivated iOS v2, then demands an Android-v2; (b) greenfield pays no rebuild tax — the delta between "parity" and "improved" is mostly *which* features get built, not rework; (c) the iOS v2 bundle already designed these features' cross-platform contracts (D5/D5a/D5c reasoning), so the hard thinking is sunk. *Scope honesty:* the release TRACK ramps — closed testing can open at configurator+output+workshop completeness while the learns gates finish — but the **production 1.0 bar carries the full set**. Alternative (iOS-parity-first, learns later) is the owner's to pick at Gate 0 if calendar-to-store matters more than completeness.

**AD1 — Where it lives: the existing `3dprintassistant-android` repo, native restart, prototype retired-in-history.** Reuse the repo (private, correct name, prototype history worth keeping); tag the prototype `webview-prototype-final`; the AG1 scaffold commit removes the WebView shell + stale web snapshot from the working tree (git history preserves everything; audit-delta §5 records what's mined). Development happens on `main` with free pushes — **greenfield has no released users to protect**, so the iOS push gate's rationale (main mirrors TestFlight state) does not transfer; pushed main = the cross-machine backup (same two-Mac reality that drove iOS D1a). Once v1.0.0 is LIVE on Play, adopt a release-gating rule analogous to iOS (`main` mirrors the released/track state) — written into the plan's release gate. **Gate 0 precondition: the owner pushes the mac-mini's 4 unpushed prototype commits (or writes them off explicitly)** — the no-push boundary is the owner's to relax, and scaffolding on a knowingly-stale origin invites divergence with invisible work.

**AD2 — Identity.** `applicationId` (permanent on Play): **`dk.mragile.printassistant`** — matches the iOS org prefix (`dk.mragile.3DPrintAssistant`); the prototype's `com.mustio.threedpa.android` is discarded (never published). Display name "3D Print Assistant". Version `1.0.0` / integer `versionCode` starting at 1, CI-incremented (mirrors the iOS build-number stamping). Owner may override the id at Gate 0 — it cannot change after the first Play upload.

**AD3 — Stack.** Kotlin + **Jetpack Compose** (Material 3 heavily themed to brand — AD7), Gradle Kotlin DSL with version catalog, single `:app` module for v1. **minSdk 29** (Android 10, 2019 — covers the overwhelming majority of active devices at near-zero Compose cost; the AG1 spike verifies the QuickJS wrapper's real floor, which its README does not state), **targetSdk 36** (proactive — the official Play requirement is currently API 35; the expected 2026-08-31 bump to 36 is re-verified at AG7; audit-delta §4.4). No Android Studio dependency: command-line SDK + Gradle wrapper (prototype-proven path); Studio optional on the mac-mini.

**AD4 — Architecture shape (mirrors iOS v2 D4, adapted).**
- `EngineCore` (QuickJS context lifecycle on a **single-thread dispatcher** — the serial-queue model iOS proved with 54 tests; exception capture with JS stacks; polyfills: `fetch` → bundled assets, `localStorage` → app-files-backed, `console` → logcat; **continuation/`CompletableDeferred`-based readiness**, no sleep-polling — don't re-import the iOS IR-5 debt) + domain-typed `EngineAPI` facades (Profile, Catalog, Troubleshooter, Tuning, Export, I18n).
- UI state: `ViewModel` + immutable UI-state (`StateFlow`/Compose state) per feature; views stay dumb; bridge calls live in ViewModels/services, never composables.
- Feature packages (`feature/configurator`, `feature/output`, `feature/workshop`, `feature/troubleshooter`, `feature/feedback`) + `designsystem/` + `enginebridge/` + `services/` — one component per file (the iOS audit's god-file lesson applied preemptively).
- Uniform error policy from day one: bridge throws typed exceptions; persistence is documented best-effort; no swallow-and-continue.
- Services rebuilt to the iOS reference semantics (not line-ported): CatalogProvider (bundled + overlay override-merge, AD8), FeedbackService (HMAC → Worker, `X-App-Source: android`), AnalyticsService (privacy shape, AD9), ReviewPrompt analog via **Play In-App Review API** (quota-managed by Play; keep the novelty-vs-eligibility separation semantics), WorkshopStore (AD5a), state persistence (codec-FIELDS snapshot, unknown-id degradation).

**AD5 — JS runtime: QuickJS via `HarlonWang/quickjs-wrapper`, vendored.** Per the audit-delta §3 research: the only option matching sync-call shape + small binary + module/polyfill support; bus-factor-1 risk mitigated by vendoring (healthy `quickjs-ng` upstream; frozen version safe for pure-logic eval). Recorded fallback: `androidx.javascriptengine` (Google-maintained, zero binary, async-only — adapter pain accepted only if QuickJS dies). **Mandatory AG1-entry spike (≤ half session, ledger-recorded GO before any UI code):** load `engine.js` + the 3 mirrored modules in the wrapper on an API-36 emulator; run a golden-mirror subset (≥5 states) + `resolveProfile`/`getWarnings`/`getFilters` round-trips; verify exception stack quality, 16 KB alignment, actual minSdk floor, init latency (budget: <150 ms to first call), polyfill viability (locales via fetch polyfill), and **which QuickJS the wrapper embeds (Bellard vs `quickjs-ng`)** — the "healthy upstream" half of the bus-factor mitigation is unverified until then. Spike failure → fallback chain: (b) `androidx.javascriptengine` with a sync-emulation adapter, then (c) re-research; each step ledger-recorded.

**AD5a — Store topology: Kotlin write-side op-log store + thin adapter into `createWorkshopTuning` (adopts iOS D5a's corrected default).** The harvest/suggestion LOGIC stays byte-identical JS (`workshop-tuning.js` + `workshop-tuning-rules.js` in the JSContext — the drift-killing choice); only store CRUD is native, contract-pinned. Kotlin `WorkshopStore`: ordered 2-space envelope emitter byte-identical to web backup output (fixture-pinned; the iOS fixture BYTES are reusable in content — verbatim web `exportJSON()` output — but live as an inline Swift string, so AG4 entry regenerates fresh web fixtures rather than copying), op-log write APIs (`addTuningOp`/`dismissSuggestion`/`revertTuning`/`addOutcome`), op-union import, atomic writes. The D5a byte evidence (web store persists COMPACT; disk envelope is PRETTY) is settled — Android simply builds the PRETTY-emitter + adapter shape from day one instead of discovering it. AG4-entry re-verifies the adapter surface against web-generated fixtures before journal-write code (cheap, ledger-recorded — the iOS spike's verification half, kept; its architecture-choice half is already decided).

**AD5b — `workshop-tuning-rules.js` freeze carries over (iOS D5b).** The W3-T7 rules→`troubleshooter.json` remedy migration stays frozen until after Android v1 ships; Android mirrors the rules file byte-identical as-is. (The migration's "second consumer" trigger is now Android rather than iOS v2 — still deferred, same churn rationale. Owner may order the web migration first at Gate 0; adds ~1 web session before AG5.)

**AD5c — Share: outbound-only via `state-codec.js` `encodeForShare` in the JS runtime (iOS D5c verbatim).** No Kotlin port of the key table or mine→safe substitution. Inbound links (App Links/assetlinks.json) are a non-goal for v1.

**AD6 — i18n: EN + DA from day one, per-gate.** Android resources (`values/` + `values-da/`), keys aligned to web `locales/*.json` naming where 1:1; language follows system locale; PARAM_LABELS stay English (standing rule); engine strings localize via `Engine.setLang`. **Greenfield advantage over iOS G6: there is no migration gate — the "every gate adds EN+DA strings" rule is active from AG2**, with a DA-copy review packet at the polish gate. No in-app language toggle in v1.

**AD7 — Design system.** Brand identity carries (dark bg `#0a0a0b`, primary green `#00e5a0`, Syne + DM Mono — both OFL, bundled); Material 3 components re-themed, not default-styled (the "generic Material look" is the named failure mode). Tokens: color/type/spacing/radius with font-scale ramps, motion tokens (spring specs, stagger interval, disable-animations fallbacks honoring the system animator-scale/remove-animations settings), haptic vocabulary. The configurator keeps the **step progress indicator** (dots + label) as a first-class element — owner-priority UX. Dark-primary; light theme is a non-goal (matches iOS).

**AD8 — Printer delta strategy: Android consumes a same-shaped sibling overlay.** New artifact `catalog/android-printer-overlay-v1.json` (same schema: `content_version` YYYYMMDDXX, `min_app_version` in ANDROID versioning, `payload_sha256`, additive brands/printers), **override-merge-by-id from day one** (no iOS Part-C all-or-nothing legacy). Web-side publisher work: generalize `validate-ios-printer-overlay.js` with a platform parameter + extend the printer-addition runbook — **AG7 precondition (Codex CX-2, run-verified): the iOS validator is currently RED (requires a bundled-catalog baseline for live iOS MARKETING_VERSION 1.0.7; baselines stop at 1.0.5) — make the existing infra green first (add the 1.0.7 baseline) or explicitly decouple the Android validator path from iOS baseline freshness**; landed as web commits at the Android release gate. Until first release, Android's bundled `printers.json` simply mirror-syncs with web per gate (greenfield — no released baseline to protect); the baseline freezes at v1.0.0 and the overlay takes over. *Rejected:* bundle-only (re-creates release-latency for printer adds; the intake automation ships printers web-first same-day and Android should ride it), consuming the iOS overlay (per-platform `min_app_version` semantics collide).

**AD9 — Analytics + feedback.**
- Analytics: same privacy shape (no user/session/device ids, no free text), HMAC pattern, `platform:"android"` (the payload validator takes any platform string — verified), `channel`: `"play"` / `"internal"` mirroring iOS's channel use. **Two prerequisite web commits before AG3** (each its own finding-commit; design-review HIGH-1, controller-verified): (1) **extend the Worker's auth branch to accept `android` as an HMAC-verified source** — today `analytics.js` HMAC-checks only `X-App-Source: ios` and 403s every other non-browser request on the origin check, so a native Android client is `forbidden_origin` before its HMAC is read (audit-delta §2.6); (2) **the mined `EVENT_KEYS` bug** — the Worker rejects `troubleshoot_used`/`export_clicked` for every platform today; fixing it repairs web's own dropped events (standalone web finding, not Android scope). **The fix is a full schema patch, not a two-line allowlist** (Codex CX-1): the events carry props the Worker has no slots for (`troubleshoot_used {symptom}`, `export_clicked {type,…}` — `BLOB_FIELDS` and the `/analytics` query surface have no `symptom`/`type` columns), so the commit must add the per-event allowed keys AND the blob/query mapping, with tests against real web payloads; a shallow allowlist would pass a bare curl while still dropping the dimensions. **5-event parity** (`app_opened`, `profile_generated`, `feedback_opened`, `troubleshoot_used`, `export_clicked`) once both land; AG3's entry check probes the LIVE endpoint (committed ≠ deployed).
- Feedback: HMAC → `/api/feedback` Worker with `X-App-Source: android` — **same auth topology, same fix shape** (audit-delta §2.7: today a native Android client is 403'd outright — the feedback would never reach Discord): extend the HMAC branch to accept `android` + first-class source rendering (Discord footer, S2 tee provenance). Own web commit before AG4; entry check probes the live Worker. Contextual prefill semantics mirror the iOS S1 `.sheet(item:)` lesson (prefill passed as an explicit argument, never ambient state).

**AD10 — Distribution + CI.**
- **Play Console personal account** — owner creates at Gate 0 ($25, government-ID verification, 2SV, DSA **non-trader declaration**; audit-delta §4). Front-loaded because it is the calendar long-pole's first domino.
- **Play App Signing** (mandatory): owner generates the upload keystore (assistant-scripted, bare commands per `feedback_bare_secret_commands_for_owner`); keystore + passwords live outside git; GitHub secrets: base64 keystore, passwords, alias, GCP service-account JSON.
- **CI:** GitHub Actions, **manual-dispatch** workflow (mirrors `testflight.yml` discipline — no auto-release on push) building AAB → **internal track** via `r0adkll/upload-google-play` (simplest maintained option; GPP is the recorded alternative). Play quirk honored: **the first upload is manual** via the Console. A separate build+instrumented-test workflow runs per push (emulator via `reactivecircus/android-emulator-runner`); if CI emulator time proves flaky, fallback = local-emulator gating + CI build-only (ledger-recorded).
- **Track ramp:** internal (dev/owner smoke) → closed testing (recruit ≥12 testers via the 3dpa Discord; 14-day continuous-engagement clock + application questionnaire) → production (first review 7–14 days). Opening the closed track "early" is only possible via the plan's **owner-triggered AG7a split** (Console setup + listing + rating + data safety + keystore + first manual upload are ALL preconditions of the track — there is no lightweight early clock); default is the full AG7 after AG6. Realistic calendar from closed-test start to live: 3–5+ weeks — tracked in the ledger, not hidden.

## 3. Quality bar + verification invariants

1. **Contract fixtures are the imported test floor (greenfield's answer to iOS's 135-test floor).** Before ANY UI gate opens: engine golden mirrors (the web-committed golden-snapshot state set through the Android bridge, value-identical), codec round-trip + unknown-id degradation (FIELDS), workshop envelope byte-compat (web-generated fixtures; PRETTY 2-space emitter), op-log semantics fixtures (op-union, clamp derivation, pairKey guard). These run green at every gate exit thereafter.
2. **Byte-identity checks at every gate entry:** mirrored files (`engine.js`, `materials.json`, `nozzles.json`, `data/rules/`, `locales/`, the 3 mirrored JS modules; `printers.json` included pre-release per AD8) byte-diff clean against web `main` at the ledger-recorded synced SHA; drift → mirror-sync as the gate's first commit.
3. **TDD RED-first** per repo convention for all new behavior, including the TDD-RED breadcrumb rules for degenerate mirror-test REDs (the cross-platform mirror convention from `3dprintassistant/CLAUDE.md` applies to Android mirrors identically).
4. **Every gate ends:** full instrumented + JVM suite green → hostile fresh-context sub-agent review of the gate diff → patch (one finding = one commit) → QA green → ledger tick → commits pushed (AD1: main is pushable pre-release).
5. **Cross-model (Codex) review** minimum at AG1 exit (architecture + bridge) and AG-release entry (full diff); `bridge --health` → `bridge --mode codex-only`, direct `codex exec -s read-only` fallback (per RUNBOOK).
6. **Accessibility acceptance per screen:** TalkBack labels/roles on all interactive elements, 200% font scale renders without clipping, animator-scale-zero (reduce motion) honored, touch targets ≥48 dp. Evidence at the polish gate = a committed per-screen checklist artifact, not adjectives.
7. **Screenshot packets** at each UI gate (emulator screenshots via Compose-test/UiAutomator harness — the ScreenCaptureUITests analog, built at AG2 and maintained by each UI gate) for async owner review; owner taste feedback queued, non-blocking, lands as patch commits.
8. **Data/logic-change evaluation (standing rule):** this program changes no engine logic and no data; the two web Worker fixes (analytics EVENT_KEYS, feedback appSource) each carry their own web+iOS impact note in their commits (web app.js already sends the events — the fix un-drops them; iOS gains the same 2 events as a FUTURE iOS train, noted not executed).
