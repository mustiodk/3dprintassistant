## Decisions SYN-00 ‚Ä¶ SYN-17

- **SYN-00 ‚Äî Local-first, accounts optional, forever.** *(Both candidates agree.)* Every shipped feature keeps working signed-out with device-local storage; signed-out UX framing is Codex's "On this device"; the promise copy is Claude ¬ß5 ("3dpa works fully without an account‚Ä¶" EN+DA). No login prompt on first launch or first profile; the conversion moment is explicit sync/backup intent (Codex ¬ß4.2).

- **SYN-01 ‚Äî Product boundary: sync = Pro (OWNER-LOCKED).** Filament inventory is **free and local-first** (unflagged additive feature, MD0-consistent). **Cloud sync + multi-device (Workshop + inventory) is the one-time "3dpa Pro" unlock** ‚Äî non-consumable IAP, suggested **99 DKK** (Codex's price research; MD2 range 49‚Äì99 kr; owner confirms at MG0). Consequences adopted from Claude APD13(b): the iOS purchase rail must exist **before** public sync launch (web users buy on iOS per MD3 ‚Äî web checkout stays deferred); the MD2 amendment note lands in the same PR that ratifies MG0. Cloud sync was already an MD2 Pro candidate, so no anchor is burned. **Never data hostage:** on entitlement absence/lapse/refund, cloud-held data stays readable, exportable, and deletable, and local data is untouched ‚Äî only sync *writes* stop (adapts Codex ¬ß12.3's non-destructive over-limit posture to the sync boundary). No spool caps exist under this boundary (Codex's 50-spool tier is retired with its Pro anchor).

- **SYN-02 ‚Äî Free-account value (owner ratifies).** A free account = identity + Pro-entitlement portability + the My 3DPA page. No free cloud data features (backup IS sync's value under SYN-01). Accounts are created only on explicit user action (sign-in offered at Pro purchase, restore purchases, or from My 3DPA) ‚Äî no anonymous/implicit account state (Codex ¬ß7.4 last paragraph).

- **SYN-03 ‚Äî Identity = Firebase Auth (OWNER-LOCKED). Adopt Codex ¬ß6‚Äì¬ß7 with these bindings:** Firebase Authentication (identity only) with **Sign in with Apple + Google**, no email/password v1; Worker verifies RS256 ID tokens against Google JWKS with Codex ¬ß7.3's fail-closed caching; `user_id` = verified Firebase `sub`; D1 `users.status='active'` check makes deletion immediate; **no email in D1**; anonymous Firebase users rejected; provider-link collision = safe-stop before mutation (no auto-merge, email never links ‚Äî both candidates agree). Web loads the Firebase SDK lazily only when account UI opens, behind a strict CSP allowlist (Codex ¬ß11.1) ‚Äî the anonymous configurator path stays third-party-free. **Carried over from Claude:** Apple's SiwA token-revocation-on-account-deletion obligation (5.1.1(v)) must be explicitly verified through the Firebase deletion path as an MG4 exit item; if Firebase does not revoke Apple tokens, a minimal Apple `/auth/revoke` step is added to the deletion saga (Claude ¬ß4.3 revocation mechanics as fallback). Owner still creates the Apple Services ID + `.p8` key (Firebase's Apple web sign-in requires them ‚Äî configured in the Firebase console, never in the repo) and enables the SiwA capability on the App ID.

- **SYN-04 ‚Äî Device model: registration yes, request signing no.** Keep Codex's device registration (`devices/register` is the sole account creator, atomic user+device transaction, random client-generated device IDs, revocation, lost-device flow = Firebase refresh-token revocation + device revocation). **Drop the P-256/ECDSA signed-request layer, monotonic-counter cryptographic binding, and cached-mutation-result machinery** ‚Äî replaced by Firebase bearer auth + per-op idempotency (`opId = deviceId:opSequence`, `UNIQUE(user_id, op_id)`, duplicate-with-same-hash returns recorded result, different bytes ‚Üí `409 idempotency_mismatch` ‚Äî Codex ¬ß9.3 semantics minus the signature). **Recorded accepted risk:** a stolen valid Firebase token can act as any of the user's devices until expiry/revocation; proportionate at 3dpa scale. This is a deliberate deviation from Codex's reviewed zero-P0 state ‚Üí SYN-17 fresh review is mandatory.

- **SYN-05 ‚Äî Backend topology.** Cloudflare Workers in the existing web repo (`functions/api/v1/**`, exact method+path router, JSON 404/405, no Assets fall-through ‚Äî Claude APD1) + **one EU-jurisdiction D1** (`--jurisdiction=eu`, checked-in numbered expand-only migrations, migrate-before-deploy ‚Äî Claude ¬ß4.2 discipline) + **KV for the account-erasure ledger** (outside the D1 restore boundary ‚Äî Claude's mechanism chosen over Codex's R2 ledger: cheaper, TTL-native, already-operated primitive). **No R2, Queues, or Durable Objects in V0‚ÄìV3**; recorded scale triggers: async export jobs (R2+Queues) when synchronous export payloads exceed ~5 MB p95, graveyard compaction (SYN-08) when tombstone rows materially move D1 cost. Flags are **Worker secrets, not `[vars]`** (Claude AP1 ‚Äî git-connected deploys re-apply `[vars]`). Rate limits per Claude APD12 (in-Worker bindings; IP-keyed pre-auth, user-keyed post-auth) with Codex ¬ß9.2's day-1 numeric ceilings as the starting configuration. Adopt Codex ¬ß6.3's cost model + 50%/75% budget alerts + global write breaker (sync goes read-only; pull/export/delete stay up).

- **SYN-06 ‚Äî Data contract: PDM2, contract-first (adopt Codex ¬ß8 + Gate C0 verbatim, trimmed kinds).** Checked-in JSON Schema 2020-12 files under `contracts/pdm2/**` + `namespaces.json` + manifest root hash + shared valid/invalid/golden fixtures + IC0 pre-merge Swift parity. Entity kinds v1-active: `profile`, `outcome`, `tuning_op`, `tuning_dismissal`, `spool`, `inventory_event`, `preference`. Defined-but-dormant (schemas exist at C0, server enable per milestone): `custom_material` (W4 train), `export_snapshot`, `export_preset` (V4), `printer` (V4). Canonical hashing = RFC 8785 + SHA-256; IDs/timestamps per Codex ¬ß8.3.

- **SYN-07 ‚Äî Sync model: entity/op-level, server-authoritative revisions, conflict copies (adopt Codex ¬ß9 with SYN-04/-08 simplifications).** Server-assigned monotonic per-user `user_revision`; push = contiguous per-device op batches with dependency ordering, per-op typed results (`applied|duplicate|conflict|rejected`), acknowledgement watermarks; pull = cursor + `currentRevision`/`minAvailableRevision`; field-mask updates + `426` schema negotiation (older client never lossy-writes newer fields); append-only kinds union; **mutable kinds get visible conflict copies (Codex ¬ß9.4), NOT client-side per-field merge** ‚Äî this deletes Claude's largest new component (the cross-platform per-field rev-map merge layer, AP5/I2) entirely. Preference/tuning_dismissal = latest-per-key with deterministic UUID v5. The server validates schemas/quotas but **never runs engine logic** (both candidates). Limits: 100 ops/push, 500 entities/page, 64 KB/entity, 5 MB active payload/account (Codex ¬ß9.2).

- **SYN-08 ‚Äî Deletion semantics: entity vs account are different guarantees.** *Entity deletes* are ordinary synced content-free tombstone ops in D1 (one transaction, versioned; stale edit after delete ‚Üí conflict with "Restore as new copy" ‚Äî Codex ¬ß9.4). **Tombstones are retained until account deletion in V1‚ÄìV3 ‚Äî no compaction, therefore no HMAC graveyard buckets, no fenced lifecycle sagas, no external entity-lifecycle ledger, no per-op decision objects, no reconciler** (Codex ¬ß8.4's machinery is retired with its trigger; content-free tombstone rows are trivial at this scale; compaction+graveyard is the recorded scale trigger). A D1 Time-Travel restore may transiently resurrect recently deleted *entities* exactly as it resurrects recent writes; active clients re-converge it via re-push (self-healing) and the policy discloses the point-in-time nature. *Account deletion* carries the hard erasure guarantee: see SYN-11. This is the second deliberate deviation from Codex's reviewed state ‚Üí SYN-17.

- **SYN-09 ‚Äî Local storage & migration: opt-in only.** **Signed-out users are never migrated** ‚Äî today's stores (`3dpa_state_v1`, Workshop store, new inventory store) keep their keys and formats forever (Claude AP5 posture). PDM2 materialization (journal un-nesting, legacy-ID UUID-v5 remap per Codex ¬ß15.1) happens **only at sync opt-in**, inside the sync adapter, preceded by Codex ¬ß4.3's verified local safety backup (receipt = distinct location + hash + readback; browser fallback = explicit download confirmation). Local partitioning across accounts uses **Claude APD14 replica namespaces verbatim** (family-iPad isolation, sign-out hides account data, tombstones never cross namespaces, account-deletion keep/remove question). The shipped v1 backup format stays importable/exportable for at least two release trains after PDM2 ships (Codex ¬ß15.4).

- **SYN-10 ‚Äî Inventory: event-sourced (adopt Codex ¬ß10.3), free, local-first.** Spool = descriptive/capacity/purchase metadata; remaining/status/location/assignment derived **only** from append-only `inventory_event` folding (typed mg invariants, checked arithmetic, never silently clamped); projections are client-side v1 (no server projection table ‚Äî nothing server-side needs derived balances under SYN-01). CSV import/export with formula-injection defense + idempotent import IDs (Codex). **Carried over from Claude APD5:** the ratified field-by-field bambuinventory mapping table, the `import_meta` opaque lossless passthrough, `material_id` keyed to the `materials.json` vocabulary (`material_display` free text never match-badges), and the export‚Üíimport‚Üíexport round-trip fixture generated from the owner's real `api.php` data. bambuinventory reuse boundary per Codex ¬ß10.3 (read-only local exporter; no PHP/MySQL/Gmail/MQTT reuse). Configurator integration app-layer only; engine/data untouched (Claude APD6).

- **SYN-11 ‚Äî Account lifecycle: export, deletion, GDPR (Claude's mechanisms sized to Firebase's saga).** *Export:* **synchronous** in v1 ‚Äî one authenticated JSON download of all server-held account data (user row, devices, entities incl. retained tombstones, entitlement state, schema manifest; labeled "cloud account export"; local backup export remains the full-device path ‚Äî Claude APD7). Codex ¬ß7.5's async signed/encrypted job pipeline is the recorded scale trigger, not v1. *Deletion:* in-app on both platforms (Apple 5.1.1(v)) + web route `/account/delete`; idempotent cross-vendor saga: (1) KV erasure intent FIRST (`del:<sha256(user_id)>`, idempotent, 90-day TTL, disclosed), (2) atomic child-first D1 domain wipe (schema-migration test fails if a user-scoped table is missing from the enumerated cascade ‚Äî Codex ¬ß7.5 step 2 discipline), (3) Firebase identity deletion + refresh-token revocation with bounded retry (incl. the SYN-03 Apple-revocation verification), (4) KV completion marker. `users.status='deleting'` locks the API from step 1. *Restore rule:* any D1 Time-Travel restore replays KV-logged account deletions after the restore point **before traffic resumes**; scripted replay tool + one rehearsal is a gate exit (Claude AP9). *Kill switch:* Claude ¬ß4.5 verbatim ‚Äî with `ACCOUNTS_API_ENABLED=false`, export/delete/status/sign-out stay reachable AND usable end-to-end (post-token-expiry test), and the client still renders the minimal rights section. *Purchases at deletion:* minimal `purchase_retention` handling per Codex ¬ß12.3 (UID-free, encrypted handle, `erase_after`).

- **SYN-12 ‚Äî Privacy posture & disclosures.** `privacy.html`'s collection sections are **rewritten, not appended** (they currently deny collecting name/email/User ID ‚Äî Claude APD8), including: Firebase Auth's **US processing** of identity data + DPA/SCC evidence recorded before production GO (Codex ¬ß19.6); D1 EU jurisdiction; KV deletion-log retention; Time-Travel/restore aging window; ToS-light + 13+ age posture (Claude APD15). iOS App Privacy label (the app's first): Name (optional), Email (in Firebase, still collected), User ID, Other User Content ‚Äî linked to identity, app functionality, no tracking (Claude AP10-I5). Play Data Safety when Android joins. Analytics anonymity unchanged: account identifiers never enter Analytics Engine; grep-level assertion ships with the first account UI gate and runs in every later PR, runtime-payload assertion at launch gate (Claude AP3/AP9 mechanism); account metrics are separate content-free counters (Codex ¬ß14.2).

- **SYN-13 ‚Äî My 3DPA hub.** Codex's naming ("My 3DPA", not "Profile") and section frame; **Claude's lean v1 contents** ship first (identity + linked providers, per-kind sync status, Workshop/inventory summaries, cloud export, delete, sign out). Sections activate per milestone: Filament (V1), Sync & Backup + Account & Privacy (V2), Exports + Printers (V4). Web = in-app view on the Workshop surface pattern; iOS = Settings-reachable screen (Claude APD7).

- **SYN-14 ‚Äî Rollout & environments.** Staging Worker + staging D1 + staging Firebase project, **Cloudflare Access-gated owner-only, corrected privacy notice pre-consent, scripted post-E2E purge** (Claude AP1 + Codex B1's Firebase-staging evidence discipline merged). Production sequence per milestone V2/V3: **privacy policy live ‚Üí prod dark (API on, UI off) ‚Üí owner canary (~48h, real accounts, both providers, two browsers, export + delete + re-auth-empty) ‚Üí single public flip.** No percentage rollout (unmeasurable at 3dpa traffic). Hard rule (Claude C14‚ÄìC16 catch): **no production collection before the rewritten policy is live.** iOS: own release train after the 1.0.7 and 1.0.8 trains, suggested `MARKETING_VERSION=1.1.0`, sub-gate commits held on a non-deploying remote feature branch, one `main` push at TestFlight-ready, ~7-day TestFlight soak before App Store release (Claude AP10 + Codex I1c, trimmed).

- **SYN-15 ‚Äî Platform sequencing.** Web ‚Üí iOS 1.1.0 (V3) ‚Üí Android v1.1 sync train after Android v1 ships under AG0 (never blocks Android v1 ‚Äî both candidates) ‚Üí macOS via shared Swift packages later (Codex ¬ß11.4). Android/macOS get pointer gates, not full briefs, in the merged plan.

- **SYN-16 ‚Äî Entitlement rail (adopt Codex ¬ß12.3 verbatim, server-authoritative).** StoreKit 2 signed-transaction JWS validation + App Store Server API reconciliation + ASSN v2 notifications (dedup by `notificationUUID`); server-issued per-account `appAccountToken` (never client-generated/logged/exported) bound at validation; Restore Purchases on every purchase surface; 7-day offline grace from last server verification; sandbox never grants production; refund/revocation applies the SYN-01 never-data-hostage state; Play Billing mirror when Android joins. Entitlement guard on sync-write endpoints only (`403 pro_required` typed error; pull/export/delete never gated).
