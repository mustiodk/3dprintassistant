## Synthesis rule

## Decisions SYN-00 … SYN-17

- **SYN-00 — Local-first, accounts optional, forever.** *(Both candidates agree.)* Every shipped feature keeps working signed-out with device-local storage; signed-out UX framing is Codex's "On this device"; the promise copy is Claude §5 ("3dpa works fully without an account…" EN+DA). No login prompt on first launch or first profile; the conversion moment is explicit sync/backup intent (Codex §4.2).

- **SYN-01 — Product boundary: sync = Pro (OWNER-LOCKED).** Filament inventory is **free and local-first** (unflagged additive feature, MD0-consistent). **Cloud sync + multi-device (Workshop + inventory) is the one-time "3dpa Pro" unlock** — non-consumable IAP, suggested **99 DKK** (Codex's price research; MD2 range 49–99 kr; owner confirms at MG0). Consequences adopted from Claude APD13(b): the iOS purchase rail must exist **before** public sync launch (web users buy on iOS per MD3 — web checkout stays deferred); the MD2 amendment note lands in the same PR that ratifies MG0. Cloud sync was already an MD2 Pro candidate, so no anchor is burned. **Never data hostage:** on entitlement absence/lapse/refund, cloud-held data stays readable, exportable, and deletable, and local data is untouched — only sync *writes* stop (adapts Codex §12.3's non-destructive over-limit posture to the sync boundary). No spool caps exist under this boundary (Codex's 50-spool tier is retired with its Pro anchor).

- **SYN-02 — Free-account value (owner ratifies).** A free account = identity + Pro-entitlement portability + the My 3DPA page. No free cloud data features (backup IS sync's value under SYN-01). Accounts are created only on explicit user action (sign-in offered at Pro purchase, restore purchases, or from My 3DPA) — no anonymous/implicit account state (Codex §7.4 last paragraph).

- **SYN-03 — Identity = Firebase Auth (OWNER-LOCKED). Adopt Codex §6–§7 with these bindings:** Firebase Authentication (identity only) with **Sign in with Apple + Google**, no email/password v1; Worker verifies RS256 ID tokens against Google JWKS with Codex §7.3's fail-closed caching; `user_id` = verified Firebase `sub`; D1 `users.status='active'` check makes deletion immediate; **no email in D1**; anonymous Firebase users rejected; provider-link collision = safe-stop before mutation (no auto-merge, email never links — both candidates agree). Web loads the Firebase SDK lazily only when account UI opens, behind a strict CSP allowlist (Codex §11.1) — the anonymous configurator path stays third-party-free. **Carried over from Claude:** Apple's SiwA token-revocation-on-account-deletion obligation (5.1.1(v)). Verification is an EARLY MG4 item with a named oracle (Apple exposes no revocation-query API): delete the staging Firebase test user, then verify the app has disappeared from the owner's appleid.apple.com "Sign in with Apple" list — before/after evidence in the ledger, run before broad staging sign-in testing. **Working assumption: Firebase user deletion does NOT auto-revoke Apple tokens**, so the fallback is planned as real scope, not a one-liner: capture Apple's `authorizationCode` at sign-in on web + iOS, Worker exchanges it (ES256 client secret from the `.p8`) and stores the encrypted grant (additive schema-amendment column), deletion saga step 3 calls Apple `/auth/revoke` — descoped only if the MG4 verification proves it unnecessary. Owner still creates the Apple Services ID + `.p8` key (Firebase's Apple web sign-in requires them — configured in the Firebase console, never in the repo) and enables the SiwA capability on the App ID.

- **SYN-04 — Device model: registration yes, request signing no.** Keep Codex's device registration (`devices/register` is the sole account creator, atomic user+device transaction, random client-generated device IDs, revocation, lost-device flow = Firebase refresh-token revocation + device revocation). **Drop the P-256/ECDSA signed-request layer, monotonic-counter cryptographic binding, and cached-mutation-result machinery** — replaced by Firebase bearer auth + per-op idempotency (`opId = deviceId:opSequence`, `UNIQUE(user_id, op_id)`, duplicate-with-same-hash returns recorded result, different bytes → `409 idempotency_mismatch` — Codex §9.3 semantics minus the signature). **Recorded accepted risk:** a stolen valid Firebase token can act as any of the user's devices until expiry/revocation; proportionate at 3dpa scale. This is a deliberate deviation from Codex's reviewed zero-P0 state → SYN-17 fresh review is mandatory.

- **SYN-05 — Backend topology.** Cloudflare Workers in the existing web repo (`functions/api/v1/**`, exact method+path router, JSON 404/405, no Assets fall-through — Claude APD1) + **one EU-jurisdiction D1** (`--jurisdiction=eu`, checked-in numbered expand-only migrations, migrate-before-deploy — Claude §4.2 discipline) + **KV for the account-erasure ledger** (outside the D1 restore boundary — Claude's mechanism chosen over Codex's R2 ledger: cheaper, TTL-native, already-operated primitive). **No R2, Queues, or Durable Objects in V0–V3**; recorded scale triggers: async export jobs (R2+Queues) when synchronous export payloads — **counting tombstone rows, which sit outside the active-payload cap** — exceed ~5 MB p95 or Worker CPU limits on the owner's recorded plan, and graveyard compaction (SYN-08) at the 50k-tombstones/account alert. Flags are **Worker secrets, not `[vars]`** (Claude AP1 — git-connected deploys re-apply `[vars]`). Rate limits per Claude APD12 (in-Worker bindings; IP-keyed pre-auth, user-keyed post-auth) with Codex §9.2's day-1 numeric ceilings as the starting configuration. Adopt Codex §6.3's cost model + 50%/75% budget alerts + global write breaker (sync goes read-only; pull/export/delete stay up).

- **SYN-06 — Data contract: PDM2, contract-first (adopt Codex §8 + Gate C0, trimmed to what survives this synthesis).** Checked-in JSON Schema 2020-12 files under `contracts/pdm2/**` + `namespaces.json` + manifest root hash + shared valid/invalid/golden fixtures + IC0 pre-merge Swift parity. Entity kinds v1-active: `profile`, `outcome`, `tuning_op`, `tuning_dismissal`, `spool`, `inventory_event`, `preference`. Defined-but-dormant (schemas exist at C0, server enable per milestone): `custom_material` (W4 train), `export_snapshot`, `export_preset` (V4), `printer` (V5 — see SYN-13). **Trim rule:** the contract contains ONLY artifacts with a surviving referent — no lifecycle-endpoint schema, no async-export-job/deletion-capability API schemas, no graveyard `hash_set_blob` vectors, no R2 conditional-create probe (all removed by SYN-04/-08/-11); the API-schema set is enumerated in MG2. `GET /api/v1/schema/versions` (the cacheable per-kind min/max read/write manifest that SYN-07's 426 discipline presupposes) is in-contract and implemented at MG3. Canonical hashing = RFC 8785 + SHA-256; IDs/timestamps per Codex §8.3.

- **SYN-07 — Sync model: entity/op-level, server-authoritative revisions, conflict copies (adopt Codex §9 with SYN-04/-08 simplifications).** Server-assigned monotonic per-user `user_revision`; push = contiguous per-device op batches with dependency ordering, per-op typed results (`applied|duplicate|conflict|rejected`), acknowledgement watermarks; pull = cursor + `currentRevision`/`minAvailableRevision`; field-mask updates + `426` schema negotiation (older client never lossy-writes newer fields); append-only kinds union; **mutable kinds get visible conflict copies (Codex §9.4), NOT client-side per-field merge** — this deletes Claude's largest new component (the cross-platform per-field rev-map merge layer, AP5/I2) entirely. Preference/tuning_dismissal = latest-per-key with deterministic UUID v5. The server validates schemas/quotas but **never runs engine logic** (both candidates). Limits: 100 ops/push, 500 entities/page, 64 KB/entity, 5 MB active payload/account (Codex §9.2). **`sync_ops` retention adopts Codex §14.3:** results are kept until client acknowledgement + 30 days, latest 1,024 acknowledged results/device retained as diagnostics, older acknowledged rows compact to the device watermark; a retry at/below the watermark returns `already_acknowledged_pull_required` and cannot mutate — so op history is bounded, not immortal.

- **SYN-08 — Deletion semantics: entity vs account are different guarantees.** *Entity deletes* are ordinary synced content-free tombstone ops in D1 (one transaction, versioned; stale edit after delete → conflict with "Restore as new copy" — Codex §9.4). **Tombstones are retained until account deletion in V1–V3 — no compaction, therefore no HMAC graveyard buckets, no fenced lifecycle sagas, no external entity-lifecycle ledger, no per-op decision objects, no reconciler** (Codex §8.4's machinery is retired with its trigger; content-free tombstone rows are trivial at this scale; compaction+graveyard is the recorded scale trigger, quantified: per-account tombstone row count joins the SYN-05 budget alerts with an alert at 50k rows/account). Consequences stated, not implied: `users.min_available_revision` is constant in V1–V3, so the `410 cursor_expired`/bootstrap-reconciliation path is contract-reserved and exercised only via injected test state (MG9/MG10); tombstone rows count toward the SYN-11 export-size scale trigger even though they are outside the 5 MB active-payload cap. A D1 Time-Travel restore may transiently resurrect recently deleted *entities* exactly as it resurrects recent writes; active clients re-converge it via re-push (self-healing) and the policy discloses the point-in-time nature. *Account deletion* carries the hard erasure guarantee: see SYN-11. This is the second deliberate deviation from Codex's reviewed state → SYN-17.

- **SYN-09 — Local storage & migration: opt-in only; v1-shaped stores remain the local source of record FOREVER.** **Signed-out users are never migrated** — today's stores (`3dpa_state_v1`, Workshop store, new inventory store) keep their keys and formats (Claude AP5 posture). **Account-namespace storage model (binding):** after sync opt-in the account namespace is ALSO a v1-shaped store (namespace-prefixed keys per APD14) — the entire existing Workshop/inventory UI keeps reading the same shapes unchanged; **PDM2 exists only at the sync boundary.** A per-namespace **sync sidecar** (`3dpa_sync_meta_<ns>`) holds what v1 shapes cannot: server entity versions/baseVersions, the stable journal-entry and legacy-ID maps (UUID-v5 remap per Codex §15.1), the outbox, the local deletion ledger/tombstones, and conflict-copy bookkeeping. Entities are derived from the v1-shaped store at push time (journal un-nesting happens in this derivation, keyed by the sidecar's stable entry-id map) and pulled changes fold back into v1 shapes. Consequences: no separate UI read path is ever built (Codex W0 stays deleted); APD14 is adopted **adapted, not verbatim** — its "fresh revs" reads as "fresh entity ids, server versions reset to 0" (per-field rev maps do not exist under SYN-07), and account-deletion "keep" needs no PDM2→v1 back-conversion (records are already v1-shaped; the sidecar is discarded and ids regenerate); a post-opt-in code revert strands only the inert sidecar + namespace prefix, never the data. Opt-in is preceded by Codex §4.3's verified local safety backup (receipt = distinct location + hash + readback; browser fallback = explicit download confirmation). Local partitioning across accounts uses Claude APD14 (family-iPad isolation, sign-out hides account data, tombstones never cross namespaces, account-deletion keep/remove question) with the adaptation above. **The shipped v1 backup format stays importable/exportable for as long as unmigrated signed-out stores exist — i.e., indefinitely** (Codex §15.4's two-train sunset presupposed mandatory migration and is not adopted).

- **SYN-10 — Inventory: event-sourced (adopt Codex §10.3), free, local-first.** Spool = descriptive/capacity/purchase metadata; remaining/status/location/assignment derived **only** from append-only `inventory_event` folding (typed mg invariants, checked arithmetic, never silently clamped); projections are client-side v1 (no server projection table — nothing server-side needs derived balances under SYN-01). CSV import/export with formula-injection defense + idempotent import IDs (Codex). **Carried over from Claude APD5:** the ratified field-by-field bambuinventory mapping table, the `import_meta` opaque lossless passthrough, `material_id` keyed to the `materials.json` vocabulary (`material_display` free text never match-badges), and the export→import→export round-trip fixture generated from the owner's real `api.php` data. bambuinventory reuse boundary per Codex §10.3 (read-only local exporter; no PHP/MySQL/Gmail/MQTT reuse). Configurator integration app-layer only; engine/data untouched (Claude APD6).

- **SYN-11 — Account lifecycle: export, deletion, GDPR (Claude's mechanisms sized to Firebase's saga).** *Export:* **synchronous** in v1 — one authenticated JSON download of all server-held account data (user row, devices, entities incl. retained tombstones, entitlement state, schema manifest; labeled "cloud account export"; local backup export remains the full-device path — Claude APD7). Codex §7.5's async signed/encrypted job pipeline is the recorded scale trigger, not v1. *Deletion:* in-app on both platforms (Apple 5.1.1(v)) + web route `/account/delete`; idempotent cross-vendor saga adopting Codex §7.5's retained-`users`-row ordering: (1) KV erasure intent FIRST (`del:<sha256(user_id)>`, idempotent, 90-day TTL, disclosed); `users.status='deleting'` locks every API from this step AND `devices/register` rejects a `deleting` sub with a typed retryable error — a still-live Firebase identity cannot mint a fresh account mid-saga; (2) atomic child-first D1 domain wipe of every user-scoped table EXCEPT the retained `users` row and its `deletion_jobs` row (schema-migration test fails if a user-scoped table is missing from the enumerated cascade — Codex §7.5 step 2 discipline); (3) Firebase identity deletion + refresh-token revocation with retry (incl. the SYN-03 Apple-revocation step); (4) only after identity deletion confirms: delete the `users` row, write the KV completion marker, compact `deletion_jobs` to a content-free 30-day receipt. **Retry primitive:** a Worker cron trigger owns saga resumption (a user who never returns still completes deletion) and the `purchase_retention.erase_after` sweeper; `deletion_jobs` is its state table. The "delete → re-auth = fresh empty account" test asserts the post-saga state. *Restore rule:* any D1 Time-Travel restore replays KV-logged account deletions after the restore point **before traffic resumes**; scripted replay tool + one rehearsal is a gate exit (Claude AP9). *Kill switch:* Claude §4.5 verbatim — with `ACCOUNTS_API_ENABLED=false`, export/delete/status/sign-out stay reachable AND usable end-to-end (post-token-expiry test), and the client still renders the minimal rights section. *Purchases at deletion:* minimal `purchase_retention` handling per Codex §12.3 (UID-free, encrypted handle, `erase_after`).

- **SYN-12 — Privacy posture & disclosures.** `privacy.html`'s collection sections are **rewritten, not appended** (they currently deny collecting name/email/User ID — Claude APD8), including: Firebase Auth's **US processing** of identity data + DPA/SCC evidence recorded before production GO (Codex §19.6); D1 EU jurisdiction; KV deletion-log retention; Time-Travel/restore aging window; ToS-light + 13+ age posture (Claude APD15). iOS App Privacy label (the app's first): Name (optional), Email (in Firebase, still collected), User ID, Other User Content — linked to identity, app functionality, no tracking (Claude AP10-I5). Play Data Safety when Android joins. Analytics anonymity unchanged: account identifiers never enter Analytics Engine; grep-level assertion ships with the first account UI gate and runs in every later PR, runtime-payload assertion at launch gate (Claude AP3/AP9 mechanism); account metrics are separate content-free counters (Codex §14.2).

- **SYN-13 — My 3DPA hub.** Codex's naming ("My 3DPA", not "Profile") and section frame; **Claude's lean v1 contents** ship first (identity + linked providers, per-kind sync status, Workshop/inventory summaries, cloud export, delete, sign out). Sections activate per milestone: Filament (V1), Sync & Backup + Account & Privacy (V2), Exports (V4), Printers (V5/backlog — no V4 gate delivers it, so it is not promised there). Web = in-app view on the Workshop surface pattern; iOS = Settings-reachable screen (Claude APD7).

- **SYN-14 — Rollout & environments.** Staging Worker + staging D1 + staging Firebase project, **Cloudflare Access-gated owner-only, corrected privacy notice pre-consent, scripted post-E2E purge** (Claude AP1 + Codex B1's Firebase-staging evidence discipline merged). Production sequence per milestone V2/V3: **privacy policy live → prod dark (API on, UI off) → owner canary (~48h, real accounts, both providers, two browsers, export + delete + re-auth-empty) → single public flip.** No percentage rollout (unmeasurable at 3dpa traffic). Hard rule (Claude C14–C16 catch): **no production collection before the rewritten policy is live.** iOS: own release train after the 1.0.7 and 1.0.8 trains, suggested `MARKETING_VERSION=1.1.0`, sub-gate commits held on a non-deploying remote feature branch, one `main` push at TestFlight-ready, ~7-day TestFlight soak before App Store release (Claude AP10 + Codex I1c, trimmed).

- **SYN-15 — Platform sequencing.** Web → iOS 1.1.0 (V3) → Android v1.1 sync train after Android v1 ships under AG0 (never blocks Android v1 — both candidates) → macOS via shared Swift packages later (Codex §11.4). Android/macOS get pointer gates, not full briefs, in the merged plan.

- **SYN-16 — Entitlement rail (server-authoritative; ENUMERATED adoption of Codex §12.3, not verbatim).** Adopted mechanics: StoreKit 2 signed-transaction JWS validation + App Store Server API reconciliation + ASSN v2 notifications (dedup by `notificationUUID`); server-issued per-account `appAccountToken` (never client-generated/logged/exported) bound at validation; entitlement derived only from verified purchase events; Restore Purchases on every purchase surface; sandbox never grants production; `purchase_events`/`purchase_retention` handling; refund/revocation applies the SYN-01 never-data-hostage state; Play Billing mirror when Android joins. **Explicitly struck from §12.3/§12.1** (overridden by owner-locked SYN-01): the free-tier list that includes "Workshop sync" free; the 50-active-spool cap, its transactional enforcement and over-limit UX; and the "Pro is specified only after inventory v1 has real usage" sequencing — under sync = Pro the rail is a V2 launch prerequisite, built before usage data exists. **Offline grace is server-side, not client-side:** when Apple reconciliation is unreachable, the server honors the last successful verification for 7 days (there is no client-enforced offline feature to grace — sync writes are server-gated). **Two registered exceptions to "never written directly", both staging/owner-scoped:** (1) a documented `wrangler d1 execute` insert on the **staging** D1 only — the interim mechanism entitling MG9/MG10's first-run tests before the grant exists; (2) the flag-gated (`PRO_OWNER_GRANT` secret) server-side grant for explicitly allowlisted account(s) — logged, revocable, built and tested at MG11, superseding (1) for staging and the sole permitted mechanism for the MG12 **production** owner canary; revoked at MG17's launch flip list, immediately after the owner's real App Store purchase provably grants the production entitlement. Entitlement guard on sync-write endpoints only (`403 pro_required` typed error; pull/export/delete never gated).
