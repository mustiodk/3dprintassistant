3. **P1 — §4.4 / AP4–AP5 — Tombstone acknowledgment is not implementable safely.** Failure: tombstones contain no server version, so `acked_version` cannot establish that a device acknowledged “past” a tombstone. AP4 also advances acknowledgments on GET, before the client durably applies the response, and defines neither compaction ownership nor enforcement of full-pull-before-push for retired devices. **Fix:** add explicit post-apply acknowledgments, tombstone introduction versions, a `safe_compact_through` contract, retired-device push rejection, and end-to-end premature-GET/retirement tests.

4. **P1 — APD14 / AP5 — Sign-out still exposes account data on shared devices.** Failure: rule 1 says signed-out uses `local`, while rule 3 and AP5 keep account X visible and writable after sign-out. 5. **P1 — APD3 / schema / AP1–AP2 — Refresh grace lacks recoverable successor state.** Failure: a replay inside 30 seconds must return the same successor, but the schema stores only token hashes and no successor pointer/encrypted token; the raw successor cannot be reconstructed. 6. **P1 — Account deletion / AP9 — Apple revocation is not re-runnable.** Failure: after a failed `/auth/revoke`, the cascade deletes the only encrypted refresh token and the user can no longer authenticate to retry. Calling the operation “idempotent and re-runnable” is false; Apple requires a valid token/code for programmatic revocation. **Fix:** atomically enqueue an encrypted revocation outbox record before deleting the identity, retry independently, and define retention/manual fallback. [Apple TN3194](https://developer.apple.com/documentation/technotes/tn3194-handling-account-deletions-and-revoking-tokens-for-sign-in-with-apple).
