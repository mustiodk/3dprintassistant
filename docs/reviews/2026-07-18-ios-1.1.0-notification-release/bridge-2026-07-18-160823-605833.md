### P1 ŌĆö Must-Fix Before Implementation Plan

**F1 ┬Ę Queue consumer: undefined behavior on opt-out race**
┬¦8.3 + ┬¦5.3 + ┬¦8.2
Fan-out creates a `pending` delivery row for `token_hash X`. User opts out before the consumer processes that row ŌĆö device row is deleted from `push_devices`. Consumer step 3 (decrypt from `push_devices`) finds no row; behavior is unspecified. An implementation would likely throw ŌåÆ retry ŌåÆ DLQ for what should be a silent skip. The spec must state: if the device row is absent for a `pending` delivery row, mark it `invalid` and continue without retry escalation.

**F2 ┬Ę APNs provider-auth errors not classified**
┬¦8.3 steps 7ŌĆō8
The spec classifies `429`/`5xx` as retryable and `410`/`BadDeviceToken`/`DeviceTokenNotForTopic` as terminal-delete. `403 Forbidden`, `InvalidProviderToken`, `ExpiredProviderToken`, and `MissingProviderToken` ŌĆö which indicate a broken JWT or revoked `.p8` key affecting the entire batch, not a single token ŌĆö are unclassified. An implementation following the spec would apply per-message retry logic, burning the entire retry budget with no progress. These need a third class: "halt the consumer batch immediately; do not mark individual tokens invalid; DLQ the batch; halt delivery until the operator resolves the key."

**F3 ┬Ę `wrangler.toml` not in `.assetsignore`**
┬¦10.2
`[assets] directory = "."` serves the repo root. `wrangler.toml` is absent from `.assetsignore`. A GET to `/wrangler.toml` falls through to `env.ASSETS.fetch()` and is served publicly today (exposes the KV namespace ID). After implementation adds D1 and Queue bindings, their IDs are also exposed. The spec's "every local secret/config pattern" guidance is too vague to catch this. Must explicitly name `wrangler.toml`, `worker.js`, and the new `migrations/` directory as `.assetsignore` additions, with a `curl /wrangler.toml` ŌåÆ non-200 gate in the public-probe check.

### P2 ŌĆö Should-Fix or Explicitly Acknowledge

**F4 ┬Ę Token rotation creates orphaned rows ŌĆö accepted trade-off not stated**
┬¦7.1 + ┬¦8.2. APNs token rotation leaves the old row until `BadDeviceToken` cleanup on next delivery; during the overlap both rows get delivery attempts and the device could receive a duplicate. The no-device-identity design makes this unavoidable. The spec should acknowledge it as a known accepted trade-off rather than leaving it implicit.

**F5 ┬Ę Wrangler Cron trigger absent from deployment changes list**
┬¦8.2. The "daily Wrangler Cron maintenance handler" requires `[triggers] crons` in `wrangler.toml` and an exported `scheduled` handler in `worker.js`. Neither is named in any deployment or wrangler.toml change list. Without it, 30-day retention never executes.

**F6 ┬Ę DLQ recovery path conflicts with cadence check**
┬¦8.3 + ┬¦13. Cadence blocks creating a new public campaign if any non-cancelled one exists in the 7-day window. A failed campaign is non-cancelled. Recovery must replay the existing `campaign_id` from the DLQ, not create a new campaign ŌĆö but the spec only says "follow the runbook" without confirming this. An implementation that tries to create a recovery campaign would be blocked by its own cadence check.

**F7 ┬Ę `PUSH_PUBLIC_SEND_ENABLED=false` does not cancel in-flight Queue work**
┬¦13. Disabling the flag stops new campaign creation; it does not drain an already-queued campaign. The spec must state: to stop active delivery, mark the campaign `cancelled` in D1, and the Queue consumer must check campaign status before each batch and exit cleanly on `cancelled`.

**F8 ┬Ę Two test cases missing from automated gates**
