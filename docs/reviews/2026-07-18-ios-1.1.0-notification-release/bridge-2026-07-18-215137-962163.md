**P1-A ┬╖ Vitest config scope unspecified тАФ `npm test` will sweep existing `node:test` files**

- **Evidence:** `functions/api/analytics.test.mjs`, `functions/api/analytics-query.test.mjs`, and `functions/api/feedback.intent.test.mjs` all open with `import test from "node:test"` / `// node --test тАж`. - **References:** Plan Task 2 Step 1; plan Task 9 Step 1; `functions/api/analytics.test.mjs` line 1.

**P1-C ┬╖ `UNUserNotificationCenter.current().delegate` assignment absent from plan**

- **Evidence:** Plan ┬з7.1 says `AppNotificationDelegate` implements both `UIApplicationDelegate` and `UNUserNotificationCenterDelegate`, wired via `@UIApplicationDelegateAdaptor`. `@UIApplicationDelegateAdaptor` covers only `UIApplicationDelegate` callbacks. `UNUserNotificationCenter.current().delegate` is a distinct property that must be explicitly assigned for the center to forward `userNotificationCenter(_:willPresent:withCompletionHandler:)` and `userNotificationCenter(_:didReceive:withCompletionHandler:)` to the delegate. Neither the interface spec (┬з7.1), Task 4 Step 4, nor Task 5 Step 3 mentions this assignment.
- **Impact:** Without `UNUserNotificationCenter.current().delegate = self`, foreground notifications are silently dropped by the system (not presented), and the tap-response callback тАФ the entire cold/warm launch routing path тАФ never fires. The canary proof in Task 10 Step 5 items 1тАУ5 would all fail, with no obvious log trail pointing to the cause.
- **Fix:** Task 5 Step 3 ("Wire `@UIApplicationDelegateAdaptor`тАж") must explicitly call out that `AppNotificationDelegate.application(_:didFinishLaunchingWithOptions:)` assigns `UNUserNotificationCenter.current().delegate = self`.
- **References:** Plan Task 5 Step 3; design ┬з7.1; spec ┬з7.3 "Foreground notifications present through the normal system alert/sound policy."

**P1-D ┬╖ `NotificationCoordinator` тЖТ `ContentView`/`AppRouter` wiring unspecified**

- **Evidence:** `ContentView.swift:30` declares `@State private var router = AppRouter()` тАФ `AppRouter` is local SwiftUI state, not accessible from outside. Plan Task 5 introduces `NotificationCoordinator.consumeWhenReady(catalog:)` to trigger navigation, but never specifies: (a) whether `NotificationCoordinator` is a singleton or injected via environment; (b) how `ContentView` observes the coordinator; (c) what SwiftUI mechanism (`onChange(of:)`, `.task(id:)`, `withObservationTracking`) drives `router.push()` from a coordinator state change.

- **References:** Plan Task 5 Step 3; `ContentView.swift:30`; plan interface spec `NotificationCoordinator.consumeWhenReady(catalog:)`.

**P1-E ┬╖ `prepareForLaunch()` conditional call in `PrintAssistantApp.swift` not specified**

- **Evidence:** Design ┬з6.2 requires: "call the existing validated `PrinterCatalogProvider.prepareForLaunch()` before taking the shared Swift/JSCore catalog snapshot" on a notification-triggered launch. Current `PrintAssistantApp.swift:39-45` takes `provider.currentPrintersData` first, initializes both services from it, then fires `refreshRemoteCacheForNextLaunch()` in a background task. The `prepareForLaunch()` doc comment (`PrinterCatalogProvider.swift:73-80`) explicitly warns: "Do not call it from app launch after the engine receives its catalog snapshot." Plan Task 5 file map lists "Modify: `PrintAssistantApp.swift`" but no step describes the conditional: "if coordinator has a pending printer destination тЖТ `await provider.prepareForLaunch()` before reading `currentPrintersData`." Task 5 Step 2 tests that this is true but the mechanism to make it true is not specified.
- **Impact:** Following the plan as written, a cold-launch notification tap on a newly announced printer would read the pre-`prepareForLaunch()` snapshot. The engine and DataService share the right snapshot (identical bytes) but that snapshot may not contain the new printer. TestFlight gate #3 would expose this only if the canary printer is not in the cached overlay.
- **Fix:** Task 5 Step 2 or 3 must specify: in `PrintAssistantApp`'s `.task`, check `NotificationCoordinator.[shared/injected].hasPendingPrinterDestination` before reading `provider.currentPrintersData`; if true, `await provider.prepareForLaunch()` first.
- **References:** Design ┬з6.2 steps 1тАУ4; `PrintAssistantApp.swift:37-48`; `PrinterCatalogProvider.swift:73-80`; plan Task 5 Step 2.

**P2-A ┬╖ `AppRoute` exhaustive switch in `ContentView` тАФ adding `notificationPrinter` case is not called out as a required edit**

Plan Task 5 Step 3 says "add `AppRoute.notificationPrinter(brandID:printerID:)` to `AppRouter` additions" but `ContentView.swift:52-62` has a `switch route` over `AppRoute` that Swift requires to be exhaustive. **P2-B ┬╖ `push_deliveries` paging index columns not named in migration**

Plan Task 2 Step 2 says "indexes for topic paging, seven-day cadence, pending delivery, and retention" without specifying columns. The consumer (Task 3 Step 4) pages "eligible token hashes in stable order" by `(campaign_id, environment)` filtered on non-terminal statuses. Without an explicit index on `(campaign_id, environment, status, token_hash)`, D1 performs a full-table scan per fan-out page. For large campaigns this is a correctness-adjacent performance floor the migration should anticipate.

**P2-C ┬╖ APNs JWT expiry test requires mock time тАФ plan provides no clock injection point**

Task 3 Step 2 requires "Cache ES256 JWTs for at most 50 minutes." The test must verify a cached token older than 50 minutes is refreshed before the next APNs call. No clock abstraction or injection point is specified in the `apns.js` interface. Without it the test either skips the refresh path entirely or sleeps 50 min, which is not viable.

**P2-D ┬╖ `NotificationTokenStore` file-protection class is a no-op on simulator**

Task 4 Step 2: "prove round-trip in a file whose protection is `.completeUntilFirstUserAuthentication`." iOS file-protection attributes are silently ignored on the simulator тАФ the file writes and reads regardless of the class. The test will pass green on simulator but protection behavior is not proven until a physical device. Plan should flag this; the Task 10 canary step on a physical device is the only real validation gate.
