## F9 patch coherence analysis

### F1 (opt-out race) × F9

F1 fix: fan-out step 3 CAS (pending/retryable → `consent_removed`) when device row disappears.
F9 fix: rotation transaction atomically marks pending/retryable delivery rows `token_rotated` **and** deletes the old device row in one D1 transaction.

Because both mutations are in a single atomic D1 commit, the consumer never observes "device row missing + delivery row still pending" as a rotation-caused state. When the consumer's CAS runs after rotation, the delivery is already `token_rotated` and the CAS condition (`status IN pending/retryable`) does not match → consumer reads `token_rotated` and skips per step 2. Pre-creation race (rotation before the delivery row exists) is also safe: the old `token_hash` disappears from `push_devices` before the fan-out page query, so it never enters the eligible set and no delivery row is created for it. **Coherent.** ✅

### F4 (rotation orphan) × F9

F4: signed rotation uses one D1 transaction — upsert new row, delete old row.
F9: same transaction additionally marks old token's pending/retryable delivery rows `token_rotated` before the delete.
Both are in the same atomic commit; F9 is additive within F4's transaction boundary. **Coherent.** ✅

### F6 (DLQ cadence conflict) × F9

F6: DLQ replay resumes the original `campaign_id`/cursor; cadence checked only on new public campaign creation.
Fan-out step 2 skips `token_rotated`. DLQ replay traverses the same cursor and fan-out logic, so `token_rotated` deliveries are skipped in replay as well. No double-send to rotated-out tokens. No new cadence entry. **Coherent.** ✅

### Retention × F9

`token_rotated` delivery rows are not deleted on rotation — they remain as historical ledger. The 30-day cron deletes delivery rows regardless of status. Campaign-level `token_rotated_count` is in `push_campaigns` (retained for operational history, no device id). No conflict with the retention rules. **Coherent.** ✅

### Idempotency × F9

A rotation retry (client re-sends same old/new tokens after a network failure): the upsert of the new device row is idempotent. The `UPDATE push_deliveries … WHERE status IN (pending, retryable)` is also idempotent — on the second attempt, the rows are already `token_rotated` and the WHERE clause matches nothing, producing a zero-row update with no state change. The `token_rotated_count` increment could fire without touching any delivery rows on a retry (the UPDATE affected 0 rows), which may marginally overcount distinct rotations. This is an implementation-detail concern below P2 for a single-owner, low-volume service, not a spec-level gap. **Coherent.** ✅
