### New Findings

**P2 — `consent_removed` delivery status conflates token rotation with genuine opt-out**

§8.3 step 3 (F1 patch) + §8.2 rotation semantics.

When a device token rotates during an active fan-out, the atomic rotation transaction deletes `push_devices[old_token_hash]` and creates `push_devices[new_token_hash]`. The fan-out consumer encounters the missing old row and, per F1, marks the delivery `consent_removed`. But the user has not removed consent — they re-registered with a new token. Future campaigns correctly reach them via the new token.

The behavioral outcome is correct: no double-send, no retry, no DLQ. The ledger entry is wrong. An operator reading aggregate `consent_removed` counts cannot distinguish genuine opt-outs from rotation events, which affects support triage and delivery analytics.

**Smallest durable correction:** In §8.3, after step 3, add one sentence: "`consent_removed` covers both explicit opt-out and rotation-displaced tokens; aggregate counts must not be read as pure opt-out counts." Alternatively, add a `token_rotated` status to the `push_deliveries` schema and set it in the rotation transaction before deleting the old `push_devices` row — the F4 atomic transaction already has the context to distinguish the case.
