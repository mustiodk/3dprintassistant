## Codex (review)

No P0 found, but the design is not build-ready. The known description of shipped merge behavior is accurate against `workshop-store.js`.

1. **P1 — APD4 / §4.4 — Device-clock LWW is neither lossless nor reliably convergent.** A device with a fast clock can overwrite later edits indefinitely; equal timestamps can produce different winners depending on merge direction. Journal union does not protect profile name/notes/state. **Fix:** use per-field deterministic revision tuples—HLC/Lamport counter + device ID—or an operation log; add clock-skew, equal-time, reversed-order, associativity, and idempotence tests.

2. **P1 — APD4 tombstones — 90-day compaction reintroduces resurrection.** A phone offline for 91 days can restore deleted profiles, outcomes, or spools, contradicting acceptance criterion 3. **Fix:** retain tombstones until all registered devices acknowledge them, support explicit device retirement, or treat long-absent devices as new replicas that must pull before pushing. Do not use elapsed time alone.

3. **P1 — APD14 — `local_only` is not an account namespace.** After A → sign-out → B → A, one boolean cannot distinguish anonymous records, A-owned synced records, and records excluded only from B. It can stop legitimate A sync or upload A’s data to B. Delete→reauth can also re-upload the supposedly deleted account. **Fix:** maintain per-record account/sync-scope metadata or separate anonymous/account replicas; specify deletion’s local-data choice and test A→B→A plus delete→reauth.

4. **P1 — Sync protocol/plan AP4–AP5 — concurrency safety is unspecified.** Two simultaneous PUTs must not both pass a read-then-write version check; a local edit during an in-flight push can also be marked clean and never uploaded. **Fix:** use one atomic conditional SQL statement (`UPDATE … WHERE version=?`, with atomic insert handling), test two genuinely concurrent writers, and implement a single-flight client coordinator with mutation generations/dirty-after-snapshot tracking.

5. **P1 — §4.3 native auth — both patched flows remain incomplete.** The custom-scheme intermediate code is an unbound bearer code: `auth_codes` has no PKCE challenge. Also, native Apple accepts an ID token but does not require/exchange Apple’s authorization grant code, so it may never obtain the refresh token required by the patched revoke-on-deletion flow. RFC 8252 requires PKCE for native authorization-code flows, and Apple directs servers to exchange the authorization code for refresh tokens. **Fix:** bind every intermediate code to PKCE + client + redirect; require the native Apple authorization code, exchange it using the App ID client identifier, and associate the Services ID with the primary iOS App ID. [RFC 8252](https://www.rfc-editor.org/info/rfc8252/), [Apple token guidance](https://developer.apple.com/documentation/signinwithapple/receiving-a-users-identity-token), [Apple web association](https://developer.apple.com/help/account/capabilities/configure-sign-in-with-apple-for-the-web/).

6. **P1 — APD3 sessions — rotation will spuriously revoke legitimate web sessions.** Two tabs or a retried refresh can use the same token concurrently; the first rotates it and the second triggers family-reuse revocation. Cookie requirements and signing/encryption key rotation are also underspecified. **Fix:** define retry/concurrency grace or idempotent successor handling, coordinate web refreshes, require `__Host-` cookies with `Path=/` and no `Domain`, and specify separate versioned keyrings plus AEAD encryption for provider tokens.

7. **P1 — Account deletion / kill switch — “immediate cascade” is not implemented by the schema.** Every foreign key lacks `ON DELETE CASCADE`; D1 enforces foreign keys, so deleting `users` first fails. Apple revocation plus multiple D1 deletes can partially complete, and `ACCOUNTS_API_ENABLED=false` disables the deletion endpoint entirely. **Fix:** add cascades or one tested atomic child-first batch, make deletion idempotent across provider failures, verify JWT users still exist, and keep export/delete reachable when sync/sign-up is killed. [D1 foreign-key behavior](https://developers.cloudflare.com/d1/sql-api/foreign-keys/).

8. **P1 — AP1 operations — runtime `CREATE TABLE IF NOT EXISTS` is not a migration system.** It cannot apply later column/index changes and makes schema mutation part of request handling. The plan also defines no isolated preview database; existing tests use plain mocks and there is no current Miniflare/`better-sqlite3` setup. **Fix:** use checked-in numbered SQL migrations with `wrangler d1 migrations apply`, production/preview/local bindings, and an explicit deploy-before-code gate. [Cloudflare D1 migrations](https://developers.cloudflare.com/d1/reference/migrations/).

9. **P1 — §4.5 recovery/privacy — Cloudflare assumptions are wrong or incomplete.** Time Travel is 7 days on Workers Free, not 30; `weur` is only a latency hint and does not guarantee EU residency; restoring can resurrect deleted accounts. **Fix:** state the actual plan/cost, create D1 with `--jurisdiction=eu`, and add a restore procedure that reapplies deletion records. Encrypt and rotate any manual exports. [Time Travel limits](https://developers.cloudflare.com/d1/reference/time-travel/), [D1 data location](https://developers.cloudflare.com/d1/configuration/data-location/).

10. **P1 — Worker routing / AP1–AP9 — new files are not routes.** The real [worker.js](/Users/mragile.io/dev/Claude/Projects/3dprintassistant/worker.js:12) statically imports and dispatches every endpoint. AP2 onward creates handlers but does not schedule router registration; unknown paths currently fall through to Assets. **Fix:** add an exact method/path router module, update it in every API gate, return JSON 404/405 inside `/api/v1`, and integration-test through `worker.fetch`, not handlers alone.

11. **P1 — APD12 / rollout plan — abuse controls and dark launch are aspirational.** No gate configures or tests rate limiting. On the intended Free WAF tier, rate limiting provides one IP-keyed rule, not several per-user rules. AP2 leaves the API flag off, yet AP9 claims it has been on since AP2. **Fix:** add Worker rate-limit bindings keyed by authenticated user and pre-auth IP, 429 tests, and an explicit AP2 API-on/UI-off production gate. [Cloudflare rate-limit availability](https://developers.cloudflare.com/waf/rate-limiting-rules/), [Workers rate-limit binding](https://developers.cloudflare.com/workers/runtime-apis/bindings/rate-limit/).

12. **P1 — APD7/APD8/AP9 privacy/export — “everything” export and App Privacy declarations are false.** A server export cannot contain `local_only`, unsynced, or over-cap data. APD8 omits stored display name and synced Workshop/inventory notes; plan I5 even drops User ID. Apple classifies these as Name, User ID, and Other User Content. The current [privacy policy](/Users/mragile.io/dev/Claude/Projects/3dprintassistant/privacy.html:156) explicitly says name/email/User ID are not collected. **Fix:** distinguish cloud-account export from a client-assembled full backup; include identities/entitlements where applicable; perform a complete privacy inventory and GDPR notice rewrite, not an accounts paragraph. [Apple App Privacy data types](https://developer.apple.com/app-store/app-privacy-details/).

13. **P2 — Sync schema/versioning — forward compatibility is unprotected.** A newer web document containing unknown fields can be decoded and re-encoded by an older native client that drops them. **Fix:** require lossless unknown-field preservation, document schema versions/minimum client versions, and reject unsafe writers rather than silently down-converting.

14. **P2 — Inventory/journal plan — `spool_id` is not scheduled into the real store.** Current [addOutcome](/Users/mragile.io/dev/Claude/Projects/3dprintassistant/workshop-store.js:186) reconstructs a fixed record and drops extra fields; AP7 lists inventory files only, while AP8 merely syncs references. **Fix:** change `workshop-store.js` in AP7, add round-trip tests, and define display behavior when the referenced spool is deleted.

15. **P2 — Internal consistency — patched decisions were not propagated cleanly.** The spec/plan still say APD0–APD12 despite APD15; plan line 5 names the retired single `ACCOUNTS_ENABLED` flag; §7 says merge “reuses shipped machinery”; “maps 1:1” is false; AP7 is declared independently buildable but sits behind AP4–AP6. **Fix:** normalize decision ranges, flag names, gate dependencies, and risk wording before ratification.

16. **P2 — APD10 entitlements — premature schema does not prevent a later migration.** It lacks revocation/status, transaction/original-purchase identity, and source-specific uniqueness. **Fix:** defer the table to the Phase-2 entitlement design or explicitly label it disposable rather than migration-preventing.
