# 3D Print Assistant

## Architecture

- **engine.js** = ALL business logic, data loading, profile resolution, warnings, export
- **app.js** = UI only — renders filters, results, tabs. NO business logic here.
- **data/** = JSON files for printers, materials, nozzles, rules
- **index.html** = HTML shell — interactive elements are JS-generated
- **style.css** = Dark/light theme, responsive layout

## Critical Rules

1. **Never merge engine.js and app.js.** engine.js is designed to become a standalone API. app.js only calls engine.js functions.
2. **Always stage ALL modified files before committing.** If a feature touches HTML structure or adds CSS classes, index.html and style.css MUST be staged too.
3. **PARAM_LABELS stay in English** — they match Bambu Studio / slicer UI, never translated.
4. **localStorage access must be wrapped in try-catch** — private browsing mode throws.
5. **IMPL-040 single-source-of-truth rule.** Any number that appears in UI chip text MUST be computed at render time from the same source `resolveProfile` reads — never hardcoded in a data field's `desc` string. Qualitative text lives in data; numeric prefixes are generated by `engine.getFilters(state)` using the same clamping / lookup paths the emission uses. When adding a new numeric chip field: (a) keep `desc` qualitative, (b) add a prefix generator in `getFilters`, (c) add an XCTest in iOS `EngineServiceTests.swift` that parses the chip number and asserts equality with `resolveProfile` output. See `docs/specs/IMPL-040-chip-desc-parity.md`.

## AI Collaboration

Canonical cross-project AI collaboration routing lives at `../ai-operating-model/docs/ai-collab.md`.
Full 3dpa product context, architecture, engine API, app state shape, and standing rules live at `docs/3dpa-context.md`.

3dpa collaboration add-ons:

- Web is master; iOS mirrors `engine.js` and the full `data/` tree byte-identical.
- Engine/data changes require explicit web + iOS impact evaluation.
- Run the web walkthrough and iOS XCTest after engine/data changes.
- One accepted review finding = one commit per platform unless the owner explicitly chooses otherwise.
- iOS `main` stays push-gated until ready for TestFlight.
- Quality over speed; review evidence is input, not permission to ship.

### TDD-RED Breadcrumb For Cross-Platform Mirror Tests

When mirroring an already-proven behavior, such as adding an iOS XCTest that
mirrors a green web walkthrough invariant, the natural RED phase can be
degenerate because the test may pass on first run. In that case, each
cross-platform mirror test-batch commit must leave an auditable breadcrumb.

Use one of these per test-batch commit:

1. **Inverted-first on one representative.** Author the first test in the batch
   with a deliberately wrong assertion, run the test, observe the failure, then
   flip to the correct assertion. Leave either a `// RED demo verified
   YYYY-MM-DD: <one-line explanation>` comment near the flipped assertion or a
   tiny pre-commit that shows the wrong assertion and is corrected by a
   follow-up commit.
2. **Skip with explicit commit-body note** when the natural RED is degenerate in
   a way that cannot be meaningfully inverted, such as retired-ID negative
   assertions. The commit body must say: `TDD-RED skipped: test-only,
   degenerate RED is expected; collective full-suite green is the proof.`

For inverted-first, a commit-body-only note is not enough; the proof must be
visible in code or `git log -p`. For skipped/degenerated RED, a commit-body note
is sufficient. Self-reported RED without a code or history artifact is the gap
to avoid.

Applies to any cross-platform mirror commit on 3dpa, including iOS mirroring
web walkthroughs or future cross-tool mirrors.
